# Token-2022 拡張機能の入り口 — pnpm での新プロジェクト立ち上げと Metadata 拡張

**日付**: 2026年5月9日
**学習内容**: 4/20 → 4/28 → 5/2 と続けて触ってきた SPL トークン (Token-2022) の **「素のトークン」** から、いよいよ **Token-2022 の真価である "拡張機能 (Token Extensions)"** に足を踏み入れる回。最初の拡張として **MetadataPointer + Metadata** を実装し、トークンに「名前・シンボル・URI・追加メタデータ」を**Mint Account 自身に焼き付ける**スタイルを学んだ。あわせて、新しいプロジェクトを **`pnpm` で初期化**する手順と、これまで使ってきた `npm init -y` との違い・メリット・デメリットを整理する。

## 0. 本日の位置づけ

ここまでの3回で「素の SPL トークンを作って動かす」サイクルは一通り完成した:

- **4/20**: Mint Account を作成（トークンの設計図）
- **4/28**: Token Account / ATA を作成（保管庫）
- **5/2**: 発行・送金・凍結・解凍・焼却・クローズ（ライフサイクル全体）

ここまでで使っていたのは Token-2022 が提供する**標準命令**だけ。実は Token-2022 の本当の強みは、**Mint Account に「拡張機能（Extensions）」を後付けで詰め込める設計**にある。代表例:

| 拡張機能 | できること |
|---|---|
| **Metadata / MetadataPointer** | トークン名・シンボル・URI・任意のキーバリューを Mint 自体に持たせる（NFT のメタデータ規格と同等の情報をオンチェーンに） |
| **TransferFee** | 送金時に自動で手数料を徴収する |
| **NonTransferable** | 譲渡不可なトークン（SBT 的な使い方） |
| **InterestBearing** | 残高が時間とともに利息で増える |
| **PermanentDelegate** | 発行体が永久に動かせる権限を持つ |
| **TransferHook** | 送金のたびに任意のプログラムを呼び出す |
| **ConfidentialTransfer** | 残高や送金額を暗号化して隠す |
| **MintCloseAuthority** | 条件を満たした Mint 自体をクローズできるようにする |

今日はこのうち最も基本かつ実用度が高い **Metadata 系の拡張** をひと組み (`MetadataPointer` と `Metadata`) 実装する。これで「自分のトークンを Phantom や Solscan で開いた時に名前とアイコンが正しく表示される」状態を作れるようになる。

そしてもう一つ、今回は **新しいフォルダ `tokenExtension/` を切ってプロジェクトをゼロから立ち上げ直す**。これまでは `npm init -y` でプロジェクトを作っていたが、今回からは **`pnpm`** に切り替える。1 章で `pnpm` 自体をしっかり説明する。

## 1. プロジェクトのセットアップ — `pnpm` を使う

### 1.1 そもそも `pnpm` とは何か

`pnpm` は **「performant npm」**（高性能な npm）の略で、`npm` / `yarn` と並ぶ **Node.js のパッケージマネージャ**の一つ。やっていることは npm と同じ「`package.json` を読んで `node_modules/` に依存パッケージを並べ、`package-lock.json` 相当のロックファイルを書き出す」というもの。**コマンドの体系もほぼ npm と同じ**で、`npm install` → `pnpm install`、`npm run dev` → `pnpm dev` のように1対1で読み替えられる。

ただし内部の**ディスク上のパッケージの置き方**が決定的に違う:

- **npm / yarn (classic)** — プロジェクトの `node_modules/` の中に、依存パッケージの**実体ファイルを毎回コピー**して並べる。プロジェクトを 10 個作れば、同じ React のファイルが 10 個コピーされる。
- **pnpm** — まず `~/.local/share/pnpm/store/`（OS により場所は異なる）に**全パッケージを1回だけ保存**しておき、各プロジェクトの `node_modules/` には**そこへの「ハードリンク」（ファイル本体は1つ、参照だけ複数）**を置く。同じ React がディスク上には**1個しか存在しない**。

この「**コンテンツアドレス可能なグローバルストア + ハードリンク**」が `pnpm` の核心。

### 1.2 `npm init -y` と `pnpm init` の違い

これまでは `npm init -y` を使ってきた:

```bash
npm init -y
```

`-y` は "全部デフォルトで yes" の意味で、**質問なしに `package.json` を即生成**する。中身は:

```json
{
  "name": "<フォルダ名>",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": { "test": "echo \"Error: no test specified\" && exit 1" },
  "keywords": [],
  "author": "",
  "license": "ISC"
}
```

一方、今回使った:

```bash
pnpm init
```

は **`-y` を付けなくても対話なしでデフォルトの `package.json` を生成する**（pnpm は最初から非対話前提）。生成される中身は npm 版とほぼ同じだが、**`devEngines.packageManager` フィールドに pnpm のバージョン**が自動で入る点が違う:

```json
{
  "name": "tokenExtension",
  "version": "1.0.0",
  ...
  "devEngines": {
    "packageManager": {
      "name": "pnpm",
      "version": "^11.0.9",
      "onFail": "download"
    }
  }
}
```

これは「このプロジェクトは pnpm v11 系で動かす想定。違う pnpm が使われたら自動でダウンロードして揃える」という宣言で、**チーム開発で「人によってパッケージマネージャが違う問題」を防ぐ**仕組み。これが `npm init -y` には無い小さな利点。

### 1.3 `pnpm` のメリット

**1. ディスク使用量が劇的に少ない**

最大の売り。同じパッケージを何プロジェクトでも入れても、ディスク上のファイル本体は1つ。Solana 開発のように「依存ツリーが巨大（`@solana/kit` だけでも依存パッケージが数百個）」かつ「学習用にプロジェクトをいくつも作る」スタイルでは、**npm だと数 GB 食うところを pnpm では数百 MB で済む**ことがざらにある。

**2. インストール速度が速い**

ファイル実体をコピーせずハードリンクを張るだけなので、`pnpm install` は `npm install` より明確に速い。CI 環境ではキャッシュと組み合わせるとさらに差が開く。

**3. 厳密な依存解決（"strict node_modules"）**

`npm` は `node_modules/` をフラットに展開するため、**`package.json` に書いていないパッケージでも `import` できてしまう**（依存パッケージの依存パッケージが偶然そこにあるため）。`pnpm` はデフォルトで「`package.json` に直接書いたパッケージしか `import` できない」**厳密な隔離**をかける。これにより「ある日依存パッケージのバージョンが上がっただけで突然 import エラーになる」事故を未然に防げる。

**4. モノレポとの相性が良い**

**モノレポ（monorepo）**とは、複数のアプリやライブラリを**別々の Git リポジトリに分けず、1 つのリポジトリの中でフォルダ（パッケージ）として並べて管理する**構成のこと。フロント用リポ・プログラム用リポのようにリポを分ける **マルチレポ**と対比される。共通の型やユーティリティをローカルパッケージとして参照しやすく、変更を横断的に追いやすい。

`pnpm-workspace.yaml` を置くと、そのリポ内のどのディレクトリを「パッケージ」として束ねるかを宣言でき、**複数パッケージをまとめたモノレポ**として `pnpm install` や相互参照が扱いやすくなる。Solana プロジェクトでは「Anchor プログラム + フロントエンド + 共通型定義」のようにフォルダを分けることが多く、ここで pnpm の workspace は強い。今回のプロジェクトにも自動的に `pnpm-workspace.yaml` が生成されている。

**5. 同じコマンドを `pnpm <script>` で呼べる**

`npm` では `npm run dev` のように `run` を挟む必要があるが、`pnpm` では **`pnpm dev` で直接 `package.json` の scripts を実行できる**（衝突する組み込みコマンドは別）。今日も最後に `pnpm tsx src/metadata.ts` のように直接呼んでいる。

### 1.4 `pnpm` のデメリット

**1. シンボリックリンク / ハードリンクに対応していないツールで稀に問題**

ごく一部の古いビルドツールやネイティブモジュールは「`node_modules/` の中はファイル実体である」という前提で動いており、pnpm のリンク構造に対応していないことがある。最近はほぼ解消されているが、過去のチュートリアル通りに動かない場合の原因になることがある。

**2. 「依存パッケージの依存パッケージ」を直接 import できない**

メリット 3 の裏返し。**npm 流の「いつのまにか使えていた間接依存」が pnpm では使えない**ため、移行時に追加で `pnpm add` が必要になる場面がある。これは本来「正しい依存宣言」を書かせる方向の制約だが、**初学者にとっては最初の戸惑いポイント**。

**3. グローバルストアが OS / ディスクをまたぐと無効**

ハードリンクは**同一ファイルシステム内**でしか張れないので、外付けディスク・別パーティション・WSL ↔ Windows のような境界をまたぐとリンクが効かず、結局コピーになってしまう。WSL で開発する場合、**プロジェクトフォルダと pnpm store を同じファイルシステム上に置く**のが定石。

**4. 既存プロジェクトが npm / yarn 前提だと混在が事故の元**

同じプロジェクトで `npm install` と `pnpm install` を混ぜて使うと、ロックファイルが二重になり依存解決が崩れる。**プロジェクト単位で1つに統一**する必要がある。

**5. 知名度・ドキュメント量はまだ npm > pnpm**

公式ドキュメント以外の入門記事はやはり npm 前提のものが多い。エラーで詰まった時の検索ヒット率は npm の方が高い。ただし Solana 公式ドキュメントは近年 pnpm 推しに寄っており、この分野では問題になりにくい。

### 1.5 `npm init -y` から `pnpm init` への切り替えサマリ

| 観点 | `npm init -y`（これまで） | `pnpm init`（今回から） |
|---|---|---|
| インストールコマンド | `npm install <pkg>` | `pnpm add <pkg>` |
| 開発専用パッケージ | `npm install -D <pkg>` | `pnpm add -D <pkg>` |
| スクリプト実行 | `npm run dev` | `pnpm dev` |
| 任意コマンド実行 | `npx <cmd>` | `pnpm <cmd>` または `pnpm exec <cmd>` |
| ロックファイル | `package-lock.json` | `pnpm-lock.yaml` |
| `node_modules/` の構造 | フラット展開・実体コピー | リンク + グローバルストア |
| 同名パッケージの重複 | プロジェクト数だけ存在 | OS 全体で1つ |
| ワークスペース | `npm workspaces`（後付け） | `pnpm-workspace.yaml`（標準） |

> **どちらが「正解」か**: 個人の小さな学習プロジェクト1つだけなら `npm` でも全く困らない。一方で、**Solana 学習のように小〜中サイズのプロジェクトを次々に立ち上げる学習スタイル**、あるいは**フロント + プログラムのモノレポ構成**を組む実務では、`pnpm` の方が体験が良い。Bootcamp が pnpm 推奨なのはこのため。

### 1.7 `package.json` に `"type": "module"` を追加

`pnpm init` で生成された `package.json` に **1行追加**する:

```json
{
  "name": "tokenExtension",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "type": "module",
  ...
}
```

`"type": "module"` を入れると、Node.js はこのプロジェクトのすべての `.js` / `.ts` を **ECMAScript Module (ESM)** として扱うようになる。

**ESM**とは、JavaScript 言語仕様（ECMAScript）で定められた**公式のモジュール形式**のこと。ファイルの先頭付近で `import` / `export` を静的に書き、**どのファイルが何に依存するか**をパーサが読み取れる。ブラウザの `<script type="module">` や、現行の bundler（Vite など）が前提にしているのもこの形式。一方、Node.js が長く使ってきた **`require` / `module.exports` ベースの CommonJS (CJS)** は別系統のモジュール方式で、`"type": "module"` が無いデフォルトでは `.js` は CJS として扱われる（歴史的経緯による二系統共存）。

この文脈での「ESM として扱う」とは、**そのプロジェクトの `.js` を CJS ではなく、上記の `import` / `export` ルールのモジュールとして解釈する**という意味。ESM では:

- `import` / `export` 構文が使える（CommonJS の `require` ではなく）
- **トップレベル `await`** が使える（=「`async function main()` で囲まなくても、ファイル直下で `await rpc.getLatestBlockhash().send()` が書ける」）

今日のスクリプト `metadata.ts` も含め、Solana 関連のコードは **`await` を多用するので、トップレベル `await` が使えると見通しが圧倒的に良くなる**。`"type": "module"` は実質ほぼ必須と思って良い。

> **入れ忘れた場合の症状**: 入れずに `await` を使うと、Node.js が `SyntaxError: Cannot use import statement outside a module` や `await is only valid in async functions and the top level bodies of modules` というエラーで止まる。最初に1回入れておく癖をつけておく。

### 3.2 RPC 接続とキーペア準備

```typescript
const rpc = createSolanaRpc("http://localhost:8899");
const rpcSubscriptions = createSolanaRpcSubscriptions("ws://localhost:8900");

const authority = await generateKeyPairSigner();

await airdropFactory({ rpc, rpcSubscriptions })({
  recipientAddress: authority.address,
  lamports: lamports(5_000_000_000n), // 5 SOL
  commitment: "confirmed",
});

const mint = await generateKeyPairSigner();
```

これは 4/20 と同じ「**RPC を作る → 鍵を作る → SOL を airdrop する → Mint 用の鍵も作る**」のパターン。違いは:

- 名前が `feePayer` ではなく **`authority`**（手数料も払うが、後で **MetadataPointer の updateAuthority** や **Mint Authority** も兼ねるので「権限者」と呼んだ方が自然）
- airdrop 量が **5 SOL**（拡張機能込みの Mint Account は普通の Mint より大きく、rent も高くなるため少し多めに）

### 3.3 拡張機能の設定オブジェクトを作る

```typescript
const metadataExtension = extension("TokenMetadata", {
  updateAuthority: some(authority.address),
  mint: mint.address,
  name: "OPOS",
  symbol: "OPS",
  uri: "https://raw.githubusercontent.com/solana-developers/opos-asset/main/assets/DeveloperPortal/metadata.json",
  additionalMetadata: new Map().set("description", "Only possible on Solana"),
});

const metadataPointerExtension = extension("MetadataPointer", {
  authority: some(authority.address),
  metadataAddress: some(mint.address),
});
```

ここが今日初登場のパターン。`extension(<拡張名>, <設定>)` で**拡張機能の設定オブジェクト**を作る。**実際にチェーンに何かを書き込むのはここではない**（後で出てくる Initialize 命令の役割）。**この時点では「サイズ計算用 + 初期化命令の引数生成用」の手元のデータを作っているだけ**。以下の小節は、**上のコードブロックと同じ順**（先に `TokenMetadata`、続けて `MetadataPointer`）で追う。

#### 3.3.1 `TokenMetadata` 拡張（本体）

| フィールド | 意味 |
|---|---|
| `updateAuthority` | メタデータを後から更新できる権限者 |
| `mint` | このメタデータが紐づく Mint アドレス |
| `name` | トークン名（例: "OPOS"） |
| `symbol` | シンボル（例: "OPS"） |
| `uri` | アイコン画像や追加情報を含む JSON のオフチェーン URL |
| `additionalMetadata` | 任意のキーバリューペア（`Map`）。今回は `description` を追加 |

> **オフチェーン URI の役割**: チェーンに直接画像を載せるとコストが高いので、**画像 URL や追加属性は外部 JSON に書いておき、その JSON の URL だけを `uri` に入れる**のが慣例。今回は Solana 公式のサンプル JSON `metadata.json` を指している。Phantom や Solscan は Mint を表示する時にこの URI を fetch して画像を表示する。

> **`some(...)` とは（ここだけ先に読む用）**
>
> **Q. なぜ `updateAuthority: authority.address` と書けないの？**  
> 文脈で言う **「API」は、Web サービスの URL を叩く HTTP API のことではない**。**`extension("TokenMetadata", { ... })` に渡すオブジェクトの各プロパティや、あわせて `@solana/kit` / `@solana-program/token-2022` が公開している関数の「引数の型＝呼び出し仕様」**を、プログラマ向けインターフェースの総称としてざっくり API と呼んでいるだけ。`updateAuthority` はその型定義上、単なるアドレスではなく **`Option<...>`**（「アドレスを**持つ**／**持たない**」のどちらかを表す箱）として用意されている。だから素の `authority.address` だけを渡すと **TypeScript の型が合わずエラー**になる。
>
> **Q. 説明文の「値があるかも無いかも」は、実行時にまだ分からないという意味？**  
> **違います。** ここで言っているのは **チェーン上の仕様として、その欄が 2 通りありうる**という話です。例として `updateAuthority` は、**「誰かが更新できる（アドレスあり）」**と **「もう誰も更新できない（権限なし）」**の両方を公式に表せます。`some(authority.address)` は前者を選んだ宣言、`none()` は後者を選んだ宣言、というイメージです。今回のサンプルは **`some` を使っているので「更新権限者はいる」**と決め打ちしています。
>
> **`some(x)` と `none()` の役割**  
> `@solana/kit` の `Option<T>` は、Rust の `Option` と同じ考え方で、**`some(x)` = 箱の中身は x**、`none()` = **箱は空（＝“なし”を意図的に選んだ）**と書き分けます。`null` だけだと「うっかり書き忘れ」なのか「わざと無効にした」のか区別しにくいので、**意図をはっきり書かせる**ために `some` / `none` になっています。
>
> **このあと**  
> 直後の `MetadataPointer` でも `authority` や `metadataAddress` に `some(...)` が出てきます（同じく「オプション扱いのフィールド」だからです）。

#### 3.3.2 `MetadataPointer` 拡張

> 「**この Mint のメタデータは、どのアカウントを見れば書いてあるか**」を指す**ポインタ**。

| フィールド | 意味 |
|---|---|
| `authority` | このポインタを後から書き換えられる権限者。`null` にすれば書き換え不能 |
| `metadataAddress` | メタデータ本体が書かれているアカウントのアドレス |

今回 `metadataAddress: some(mint.address)` としている = **「メタデータは Mint Account 自身に書いてある」**と宣言している。これが Token-2022 の優れた設計で、別アカウントを用意せず **1個の Mint Account の中にメタデータも詰め込める**。

### 3.4 アカウントサイズと Rent の計算

```typescript
const spaceWithoutTokenMetadataExtension = BigInt(
  getMintSize([metadataPointerExtension]),
);

const spaceWithTokenMetadataExtension = BigInt(
  getMintSize([metadataPointerExtension, metadataExtension]),
);

const rent = await rpc
  .getMinimumBalanceForRentExemption(spaceWithTokenMetadataExtension)
  .send();
```

ここは**同じ Mint でも「今のバイト数」と「完成後のバイト数」が違う**ために、変数が二つに分かれている。

| 変数 | `getMintSize` に渡す拡張 | 主な用途 |
|------|-------------------------|----------|
| `spaceWithoutTokenMetadataExtension` | `MetadataPointer` のみ | `CreateAccount` の **`space`**（作る瞬間のデータ長） |
| `spaceWithTokenMetadataExtension` | `MetadataPointer` + `TokenMetadata` | **`rent` 計算**に使うバイト数（完成後のデータ長） |
| `rent` | （上の「完成後」バイト数を RPC に渡して算出） | `CreateAccount` の **`lamports`** |

**背景（一行）:** `getMintSize([...])` に渡す拡張の組み合わせで必要バイト数が変わる。`MetadataPointer` は固定、`TokenMetadata` は `name` などで可変なので、**CreateAccount ではまだ短い長さで作り、あとから `InitializeTokenMetadata` が領域を伸ばす**流れになる。

#### 3.4.1 なぜ `spaceWithout...` が別に要るのか（いちばんつまずきやすいところ）

> **用語:** `realloc` は、アカウント作成**後**に **データ領域のバイト数を増やす**操作。可変の `TokenMetadata` 用に、`InitializeTokenMetadata` が内部で行う。

**ポイントは「CreateAccount が決めるもの」と「あとで命令が足すもの」が違うこと。**

1. **`CreateAccount` の `space` は「今この瞬間のアカウントのデータ長（バイト数）」**  
   システムプログラムが「この長さの空き領域を作る」だけ。ここではまだ **Token-2022 のプログラムがメタデータ本文をレイアウトした後**ではない。公式サンプルでは、**この時点で含めてよい／含める前提になっているのは `MetadataPointer` まで**で、`TokenMetadata` の可変部分は **まだアカウント上に存在しない**（これから `InitializeTokenMetadata` が `realloc` で広げる）。

2. **だから `getMintSize([metadataPointerExtension])` だけを `CreateAccount` に渡す**  
   「今作る Mint のバイト数」= **Pointer 込みの小さい方**に合わせる。ここでいきなり `getMintSize([..., metadataExtension])` の**最終サイズ**で `CreateAccount` すると、**まだ中身が初期化されていないのに長いアカウントだけ先に作る**ことになり、Token-2022 が想定する **「作る → 初期化命令の列で段階的に育てる」**の流れと噛み合わない（サンプルはそう書かれていない）。

3. **`lamports: rent` と `space` が食い違うように見える件**  
   表のとおり `rent` は **完成後のバイト数**から取る。`realloc` で長くなったあとも rent 免除のままいられるよう、**伸びた後に必要な分の SOL を最初から Mint に載せておく**。データ長はまだ短いので、`space` だけが小さい、という整理になる（§3.4 の表と同じ話を命令の視点から言い直しただけ）。

**覚え方:** `spaceWithout...` = **CreateAccount が今すぐ確保する床面積**。`spaceWith...` + `rent` = **完成後の家に必要な敷金を、最初から財布に入れておく額**。床は後で増築するが、お金は先に足りる分だけ用意しておく、というイメージ。

### 3.5 Mint 作成命令（CreateAccount + InitializeMetadataPointer + InitializeMint + InitializeMetadata）

ここから命令を**4つ**続けて組み立てる。**順序が決定的に重要**。

#### 3.5.1 `CreateAccount`（システム命令）

```typescript
const createMintAccountInstruction = getCreateAccountInstruction({
  payer: authority,
  newAccount: mint,
  lamports: rent,
  space: spaceWithoutTokenMetadataExtension,
  programAddress: TOKEN_2022_PROGRAM_ADDRESS,
});
```

4/20 と同じ「Mint Account の枠を確保する」命令。違いは:

- **`space` には `MetadataPointer` だけ含めたサイズ**を渡す（メタデータ本体は後から `realloc`）
- **`lamports` には全拡張込みの rent**を渡す（最終サイズ分の rent-exempt を最初から積む）

#### 3.5.2 `InitializeMetadataPointer`（拡張の初期化）

```typescript
const initializeMetadataPointerInstruction =
  getInitializeMetadataPointerInstruction({
    mint: mint.address,
    authority: authority.address,
    metadataAddress: mint.address,
  });
```

「この Mint のメタデータは Mint Account 自身を見ろ」というポインタを書き込む。**`InitializeMint` よりも前**に呼ぶ必要がある（Token-2022 の仕様で、拡張の初期化は Mint 初期化の前）。

#### 3.5.3 `InitializeMint`（4/20 と同じ）

```typescript
const initializeMintInstruction = getInitializeMintInstruction({
  mint: mint.address,
  decimals: 9,
  mintAuthority: authority.address,
  freezeAuthority: authority.address,
});
```

通常の Mint 初期化。**ここで Mint 自体の中身（decimals, mintAuthority, freezeAuthority）が確定する**。これ以降は Mint そのものの基本フィールドは変えられない。

#### 3.5.4 `InitializeTokenMetadata`（メタデータ本体の書き込み）

```typescript
const initializeMetadataInstruction = getInitializeTokenMetadataInstruction({
  metadata: mint.address,
  updateAuthority: authority.address,
  mint: mint.address,
  mintAuthority: authority,
  name: "OPOS",
  symbol: "OPS",
  uri: "https://raw.githubusercontent.com/solana-developers/opos-asset/main/assets/DeveloperPortal/metadata.json",
});
```

`MetadataPointer` で「メタデータは Mint 自身を見ろ」と宣言してあるので、**`metadata` 引数も `mint.address`**。ここで `name` / `symbol` / `uri` がチェーンに刻まれる。

> **`mintAuthority: authority`（Signer）と `mintAuthority: authority.address`（アドレス）の使い分け**: `InitializeMint` の方は Mint を新規初期化するだけで `mintAuthority` を**設定値として保存する**ので**アドレス文字列**で良い。一方 `InitializeTokenMetadata` は **「現在の mintAuthority がメタデータの書き込みに同意した」という署名**を要求するので**Signer オブジェクト**を渡す。同じ `authority` 変数だが、用途で渡し方が変わる。

#### 3.5.5 順序の確定

つまりこの4命令の順序は:

```
CreateAccount → InitializeMetadataPointer → InitializeMint → InitializeTokenMetadata
```

> **なぜこの順序か**: ① まず空のアカウント枠が必要 ② 次に「拡張あり Mint」として認識させる拡張ポインタを書く ③ そのうえで Mint 本体を初期化 ④ 最後にポインタが指す先（= Mint 自身）にメタデータを書き込む。**順序を変えるとどこかで「まだその拡張が無効」「まだ Mint が初期化されていない」というエラーで失敗する**。

### 3.6 Token Account の作成と Mint To

メタデータの実装は 3.5 までで完結している。残りは「**実際に発行できるか確認するために Token Account を1つ作って 1,000 トークンを Mint To する**」というデモ用の追加処理。

```typescript
const tokenAccount = await generateKeyPairSigner();

const tokenAccountLen = BigInt(getTokenSize([]));

const tokenAccountRent = await rpc
  .getMinimumBalanceForRentExemption(tokenAccountLen)
  .send();

const createTokenAccountInstruction = getCreateAccountInstruction({
  payer: authority,
  newAccount: tokenAccount,
  lamports: tokenAccountRent,
  space: tokenAccountLen,
  programAddress: TOKEN_2022_PROGRAM_ADDRESS,
});

const initializeTokenAccountInstruction = getInitializeAccountInstruction({
  account: tokenAccount.address,
  mint: mint.address,
  owner: authority.address,
});
```

これは 4/28 で書いた **「ランダムキーペア方式の Token Account」** とほぼ同じ。違いは:

- **`getInitializeAccount2Instruction` ではなく `getInitializeAccountInstruction`** を使っている（`Account` 版は鍵を引数で渡す形、`Account2` 版はバイナリにベタ書きする形。挙動はほぼ同じで好みの差）
- **拡張なしの Token Account なので `getTokenSize([])` の引数は空配列**（Token Account 側に拡張を入れることもできるが、今回は素のまま）

そして MintTo:

```typescript
const mintAmount = 1_000_000_000_000n; // 1,000 tokens

const mintToInstruction = getMintToInstruction({
  mint: mint.address,
  token: tokenAccount.address,
  mintAuthority: authority,
  amount: mintAmount,
});
```

5/2 と同じ Mint To。**1,000 トークン × decimals=9** = `1_000_000_000_000n`。

### 3.7 7命令を全部1つのトランザクションに詰めて送る

```typescript
const instructions = [
  createMintAccountInstruction,
  initializeMetadataPointerInstruction,
  initializeMintInstruction,
  initializeMetadataInstruction,
  createTokenAccountInstruction,
  initializeTokenAccountInstruction,
  mintToInstruction,
];

const { value: latestBlockhash } = await rpc.getLatestBlockhash().send();

const transactionMessage = pipe(
  createTransactionMessage({ version: 0 }),
  (tx) => setTransactionMessageFeePayerSigner(authority, tx),
  (tx) => setTransactionMessageLifetimeUsingBlockhash(latestBlockhash, tx),
  (tx) => appendTransactionMessageInstructions(instructions, tx),
);

const signedTransaction =
  await signTransactionMessageWithSigners(transactionMessage);

assertIsTransactionWithBlockhashLifetime(signedTransaction);

await sendAndConfirmTransactionFactory({ rpc, rpcSubscriptions })(
  signedTransaction,
  { commitment: "confirmed", skipPreflight: true },
);
```

ここで5/2 までと違う重要ポイント:

**1. 7命令が全部1トランザクションに収まる**

Solana のトランザクションは **「1個か複数の命令を含み、すべてアトミックに成功または失敗する」**。今回は次の7命令を1トランザクションで投げている:

1. CreateAccount（Mint）
2. InitializeMetadataPointer
3. InitializeMint
4. InitializeTokenMetadata
5. CreateAccount（Token Account）
6. InitializeAccount
7. MintTo

**これらは全部成功するか全部失敗するか**。途中で失敗すれば「Mint だけ作って Token Account ができていない」みたいな中途半端な状態にはならない。これが Solana のトランザクションの強み。

> **トランザクションサイズの上限**: Solana のトランザクションは 1,232 バイトまで。今回はギリギリ収まっているが、拡張をたくさん使うとここに引っかかる。その場合は `address-lookup-tables` で住所を圧縮するか、複数トランザクションに分けるか、`createAccountWithSeed` 系を使う。

**2. `assertIsTransactionWithBlockhashLifetime` の登場**

5/2 まではこれを**型アサーション**（`as typeof signedTx & {...}`）で済ませていたが、今回は **`assertIsTransactionWithBlockhashLifetime(signedTransaction)`** という**ランタイムチェック関数**を使っている。

| 方式 | 動き |
|---|---|
| 型アサーション (`as ...`) | TypeScript の型を**むりやり**書き換える。実行時には何もしない |
| `assertIs...` | **実行時に「本当にこの型条件を満たすか」をチェックし**、満たさなければ例外を投げる。同時に TypeScript の型も絞り込まれる |

`assertIs...` の方が**安全**（嘘をつかない）かつ**シンプル**（追加の型コードが要らない）。`@solana/kit` 公式が推奨する書き方はこちら。**5/2 までの型アサーションは過渡的な書き方で、今後はこちらに統一していく**。

**3. `skipPreflight: true`**

`sendAndConfirmTransaction` のオプションに **`skipPreflight: true`** が入っている。

| オプション | 意味 |
|---|---|
| `skipPreflight: false`（デフォルト） | RPC ノードが**送信前にシミュレート**して、明らかに失敗するなら早期に弾く |
| `skipPreflight: true` | シミュレートを**スキップ**し、いきなり本送信する |

通常は `false` のままで良いが、**Token-2022 拡張のような新しめの命令はシミュレータが対応していないと「シミュレート段階で失敗扱い」になることがある**ため、ローカル開発では `true` にすることがある。ただし本番ではエラー検知のためにも `false` のままが望ましい。

## 6. 今回学んだキーコンセプト

1. **`pnpm` は npm の上位互換に近い体験を提供するパッケージマネージャ**
   - グローバルストア + ハードリンクでディスク使用量を激減
   - 厳密な依存解決でフラット展開の事故を防止
   - モノレポ標準対応・コマンドが短い（`pnpm dev`）
   - 一方で「リンク非対応の古いツール」「間接依存を直接 import できない」などの注意点はある

2. **Token-2022 の拡張機能は "Mint Account にオプションを焼き付ける" 設計**
   - 通常の `getMintSize()` ではなく **`getMintSize([extensions])`** で拡張ごとのサイズを計算する
   - **可変サイズ拡張は最初小さく作って後から自動 realloc** される。rent は最終サイズで積む
   - 拡張の初期化命令は **`InitializeMint` の前**に呼ぶ

3. **`MetadataPointer` + `TokenMetadata` のペアで「Mint 自身がメタデータを持つ」**
   - 別アカウントを使わず1個の Mint で完結できるのが Token-2022 の優位性
   - `name` / `symbol` / `uri` / `additionalMetadata` を直接書き込める
   - 画像など重いデータは外部 JSON に逃して `uri` で参照する慣例

4. **複数命令を1トランザクションにまとめて "アトミック" に実行**
   - 今日は7命令を一気に投げて、全部成功 or 全部失敗
   - Solana の根幹的な強み。サイズ上限 1,232 バイトに注意

5. **`Option<T>` と `some(...)`**
   - `null` を直接使うのではなく `some(value)` / `none()` で「値あり / なし」を型レベルで区別
   - 「うっかり書き忘れ」と「意図的に空」を分けられる

6. **`assertIsTransactionWithBlockhashLifetime` でランタイム検証**
   - 5/2 まで使っていた「型アサーション」より安全な書き方
   - 今後はこちらに統一

7. **Solana 公式 Explorer はローカルバリデータも見れる**
   - URL クエリに `cluster=custom&customUrl=...` を付けるだけ
   - Mint のページに拡張・メタデータ・所有者などが視覚的に表示される

```
[getMintSize で2種類のサイズを計算（pointer のみ / 全拡張込み）]
            ↓
[全拡張込みサイズで rent を計算]
            ↓
[CreateAccount(Mint) → InitializeMetadataPointer → InitializeMint → InitializeTokenMetadata]
            ↓
[CreateAccount(TokenAccount) → InitializeAccount → MintTo(1000)]
            ↓
[7命令を1トランザクションにまとめて pipe で組み立て → 署名 → assert → send]
            ↓
[Explorer URL を3つ console.log（TX / Mint / TokenAccount）]
            ↓
─────────── 実行 ───────────
            ↓
[pnpm tsx src/metadata.ts] → ブラウザで Mint を開くと "OPOS (OPS)" として表示
```

## 8. まとめ

今回は **「素の SPL トークン」 → 「Token-2022 拡張機能」** へとステージを上げる橋渡しの回となった。

```
[素の SPL トークンのライフサイクル (4/20 - 5/2)]
              ↓
[Token Extensions = Mint Account に "機能" を追加できる仕組み]
              ↓
[MetadataPointer + TokenMetadata で「名前・シンボル・URI」を持つ Mint]
              ↓
[Phantom や Solscan で人間にも識別可能なトークンが完成]
```

技術面では「**`getMintSize([...])` で拡張のサイズを計算する**」「**`extension(...)` で設定を作る**」「**初期化命令を Mint 初期化の前に呼ぶ**」という Token Extensions 共通の作法を体得できた。これが分かれば、今後 `TransferFee`、`InterestBearing`、`PermanentDelegate` など他の拡張に進むときも**同じパターンの繰り返し**で実装できる。

開発環境面では、**`pnpm` への切り替え**が今日の地味だが大きな転換点。学習プロジェクトを次々作る Bootcamp 形式と相性が良いだけでなく、後々フロントエンドや Anchor プログラムをモノレポで束ねる時にも自然につながる。**「`npm install` を `pnpm add` と読み替えるだけ」で 95% は移行完了**するので、最初の小さな違和感さえ越えれば移行コストは低い。

「**Solana 開発はパッケージマネージャを `pnpm` に替え、Token-2022 の `extension()` ヘルパーを覚えれば、たいていの拡張機能は組み立てられる**」というのが今日いちばん強い実感。次回以降は `TransferFee` や `MintCloseAuthority` など、**今日のテンプレ + 設定オブジェクトの差し替え**で他の拡張機能を順に試していくフェーズに入っていく。
