# トークンのライフサイクル — Mint, Transfer, Freeze/Thaw, Burn, Close

**日付**: 2026年5月2日
**学習内容**: 4/28 までに作った Mint と ATA を土台にして、SPL トークンの**実用的なライフサイクル操作**を一通り実装した。具体的には **発行（Mint To）→ 送金（Transfer）→ 凍結（Freeze）→ 凍結試験用の Transfer 失敗確認 → 解凍（Thaw）→ 焼却（Burn）→ アカウントクローズ（Close）**。Token-2022 がすでに提供している標準命令を `@solana/kit` から呼ぶスタイルは前回までと完全に同じで、命令名と引数だけが変わっていく構造を体得した。

## 0. 本日の位置づけ

ここまでの2回で土台はそろった:

- **4/20**: トークンの**設計図**である **Mint Account** を作成
- **4/28**: そのトークンを**保管する財布**として **Token Account** と **Associated Token Account（ATA）** を作成

つまり「種類の宣言」と「保管庫の用意」が終わっている状態。今日はその上で **「実際にトークンを動かす」操作群** を全部触る。これで SPL トークンを使ったプロダクトを作る上で必要な**基本動詞**は出揃うことになる。

| 操作 | 何をするか | 誰の権限が必要か |
|---|---|---|
| **Mint To** | 指定の Token Account / ATA に新規トークンを発行する（= supply を増やす） | **Mint Authority**（Mint 作成時に設定） |
| **Transfer** | ある Token Account / ATA から別の Token Account / ATA へトークンを移す | **保有者**（Token Account の `owner` フィールド） |
| **Freeze** | Token Account を凍結し、Transfer も Burn もできない状態にする | **Freeze Authority**（Mint 作成時に設定） |
| **Thaw** | 凍結を解除する | **Freeze Authority**（同上） |
| **Burn** | Token Account の残高を**焼却**して supply を減らす | **保有者** |
| **Close Account** | 残高ゼロの Token Account を閉じ、**rent を回収**する | **保有者**（Close Authority が設定されていればそれ） |

**「権限を持つ人 → 命令を発行 → Token-2022 が実行する」** という構造はすべての操作で同じ。違うのは命令名と引数だけ、というのが今日いちばん強く実感できる部分である。

## 1. 前提: スクリプトの状態

今日は新規プロジェクトを作らず、4/28 まで書いてきた **`new-token/new-token.ts` に追記**していく。前回までで以下が完成している:

```
[feePayer 鍵ペア生成] → [1 SOL airdrop]
                              ↓
[mint 鍵ペア生成] → [Mint 作成 TX] (4/20)
                              ↓
[Token Account 作成 TX]  (4/28 前半)
                              ↓
[ATA アドレスを PDA で導出] → [ATA 作成 TX] (4/28 後半)
                              ↓
                    ★ ここから今日（5/2） ★
```

> **使うのは ATA の方**: 4/28 では「学習目的で」ランダムキーペア方式の Token Account と ATA の**両方**を作ったが、今日の Mint To 以降の操作は**全部 ATA を対象**に進める。実務でユーザー口座として使うのは ATA であり、ランダム方式の Token Account はもう登場しない（4/28 の 8.3「迷ったら ATA」の方針に沿う）。

## 2. インポートに追加していく関数

今日追加する6つの操作はすべて Token-2022 が提供する命令で、`@solana-program/token-2022` から関数を1つずつインポートしていく。最終形はこうなる:

```typescript
import {
  getInitializeMintInstruction,
  getInitializeAccount2Instruction,
  getCreateAssociatedTokenInstructionAsync,
  getMintToInstruction,            // ★追加: 発行
  getTransferInstruction,          // ★追加: 送金
  getFreezeAccountInstruction,     // ★追加: 凍結
  getThawAccountInstruction,       // ★追加: 解凍
  getBurnCheckedInstruction,       // ★追加: 焼却（decimalsチェック付き）
  getCloseAccountInstruction,      // ★追加: アカウントクローズ
  getMintSize,
  getTokenSize,
  TOKEN_2022_PROGRAM_ADDRESS,
  findAssociatedTokenPda,
  fetchToken,                      // ★追加: Token Account の中身を読む
} from "@solana-program/token-2022";
```

| 追加関数 | 役割 |
|---|---|
| `getMintToInstruction` | 指定の Token Account に新規トークンを**鋳造**する命令を作る |
| `getTransferInstruction` | あるアカウントから別のアカウントへトークンを**移動**する命令を作る |
| `getFreezeAccountInstruction` | 指定の Token Account を**凍結**する命令を作る |
| `getThawAccountInstruction` | 凍結を**解除**する命令を作る |
| `getBurnCheckedInstruction` | 指定の Token Account のトークンを**焼却**する命令を作る（`Checked` 版は decimals を一緒に渡して安全性を高める） |
| `getCloseAccountInstruction` | 残高ゼロの Token Account を**クローズ**して rent を回収する命令を作る |
| `fetchToken` | Token Account の中身（残高など）を RPC から読む補助関数 |

> **`getBurnCheckedInstruction` の "Checked" は何？**: SPL Token / Token-2022 には `Burn` と `BurnChecked` の2つの命令があり、後者は**呼び出し側が渡した decimals が、Mint の decimals と一致するかをチェックする**バリデーションが入っている。誤ったMintを burn しようとした事故を防げるので、**新しいコードは原則 `Checked` 版を使う**のが推奨。同様に `Transfer` と `TransferChecked`、`MintTo` と `MintToChecked` のペアもある。今日は `Mint To` と `Transfer` は `Checked` を付けない版を使ったが、Burn だけは `Checked` を選んだ（動画の方針に合わせている）。

## 3. 準備: Mint に Freeze Authority を追加する

ここが今日いちばん見落としやすいポイント。後半の **Freeze / Thaw を使うには、Mint 作成時点で `freezeAuthority` フィールドを設定しておく必要がある**。これは後付けできない（Mint の仕様としてそうなっている）。

そのため、**4/20 で書いた `getInitializeMintInstruction` の引数を1行追記する**:

```typescript
const initializeMintInstruction = getInitializeMintInstruction({
  mint: mint.address,
  decimals: 9,
  mintAuthority: feePayer.address,
  freezeAuthority: feePayer.address,   // ★追加
});
```

| 引数 | 意味 |
|---|---|
| `mintAuthority` | **新規発行（Mint To）**を許可する鍵。指定された人しか supply を増やせない |
| `freezeAuthority` | **凍結 / 解凍（Freeze / Thaw）**を許可する鍵。指定された人しか Token Account を凍結・解除できない |

両方とも `feePayer.address` にしているので、今回のスクリプト内では feePayer がすべての権限を握っている。実務では Mint Authority と Freeze Authority を別々のマルチシグや別組織に分けることもある。

> **`freezeAuthority` を `null` にすると**: Token-2022 では `freezeAuthority` を `null`（または未指定）で初期化すると、**この Mint の全 Token Account は永久に凍結できなくなる**。USDC のような中央集権系トークンは `freezeAuthority` を保持する一方、コミュニティ系トークンや「中央が触れない」ことを保証したいトークンは意図的に `null` にする。**一度設定した authority を後から `null` にする**（権限放棄）はできるが、**一度 `null` にしたものを後から復活はできない**。

> **`mintAuthority` も同様**: `mintAuthority` を `null` にすると以降は誰も追加発行できず、**supply は固定**される。BTC 風に「総発行量を二度と増やさない」ことを明示する手段として使う。

## 4. トークンの発行（Mint To）

ATA はあるが残高は0、という状態に**実際にトークンを発行する**。

### 4.1 Mint To 命令の組み立て

```typescript
const mintToInstruction = getMintToInstruction({
  mint: mint.address,
  token: associatedTokenAccountAddress,
  mintAuthority: feePayer.address,
  amount: 1_000_000_000n,
});
```

引数の意味:

- **`mint: mint.address`**: どの Mint（= どのトークン種別）を発行するか
- **`token: associatedTokenAccountAddress`**: **発行先の Token Account**（4/28 で作った feePayer の ATA）
- **`mintAuthority: feePayer.address`**: **発行命令を出す権限者**。3 で `freezeAuthority` と一緒に Mint 作成時に設定した `mintAuthority` と一致する必要がある
- **`amount: 1_000_000_000n`**: **発行量（最小単位）**。Mint の `decimals: 9` なので 10^9 = 1 トークン

> **量は常に「最小単位」で渡す**: Solana の Token-2022 命令では量は**整数**（最小単位）で受け取る。decimals=9 のトークンを1トークン発行したければ `1_000_000_000n` を渡す。decimals=6 のトークン（USDC 等）を1トークン発行したければ `1_000_000n`。**表示用の小数値（"1.5 トークン"）は人間向けの見せ方**であり、内部では常に整数で扱う。

> **`amount: 1_000_000_000n` の `n`**: 4/20 と同じく `BigInt` リテラルの末尾 `n`。Solana 系の数量は `bigint` 統一が `@solana/kit` の流儀。

### 4.2 トランザクションを組み立てて送信

```typescript
const { value: mintToLatestBlockhash } = await rpc.getLatestBlockhash().send();

const mintToTxMessage = pipe(
  createTransactionMessage({ version: 0 }),
  (tx) => setTransactionMessageFeePayerSigner(feePayer, tx),
  (tx) =>
    setTransactionMessageLifetimeUsingBlockhash(mintToLatestBlockhash, tx),
  (tx) => appendTransactionMessageInstructions([mintToInstruction], tx),
);

const signedMintToTxMessage =
  await signTransactionMessageWithSigners(mintToTxMessage);

const signedMintToTxMessageWithLifetime =
  signedMintToTxMessage as typeof signedMintToTxMessage & {
    lifetimeConstraint: {
      lastValidBlockHeight: bigint;
    };
  };

await sendAndConfirmTransactionFactory({ rpc, rpcSubscriptions })(
  signedMintToTxMessageWithLifetime,
  { commitment: "confirmed" },
);

const mintToTxSignature = getSignatureFromTransaction(
  signedMintToTxMessageWithLifetime,
);

console.log("\nMint To Transaction Signature:", mintToTxSignature);
```

ここから後はこのパターンの**完全な繰り返し**になる。命令を1個（または複数）作る → blockhash を取る → `pipe` でメッセージを組み立てる → 署名 → 型アサーション → 送信、というテンプレ。**毎回違うのは命令だけ**で、土台はずっと同じ。

> **署名者**: Mint To 命令には `mintAuthority` の署名が必要。`feePayer = mintAuthority` なので、`feePayer` の署名だけで通る。`@solana/kit` は命令の中で「署名者として必要な鍵」を解析して `signTransactionMessageWithSigners` で自動的に集めてくれるので、明示的に署名者リストを渡す必要はない。

### 4.3 残高の確認

発行が終わったら、本当に1トークン入ったかを RPC から読む:

```typescript

  commitment: "confirmed",
});

const ataBalance = ataData.data.amount;

console.log(
  "Associated Token Account Balance:",
  Number(ataBalance) / 1_000_000_000,
);
```

- **`fetchToken(rpc, address, options)`**: Token Account の中身を RPC で取得し、`.data.amount` などにアクセスできる形にデコードしてくれる**便利関数**。生の `getAccountInfo` を呼んで自分でバイト列をパースしてもいいが、`@solana-program/token-2022` 側でこれを用意してくれている。
- **`ataData.data.amount`**: Token Account のデータ部の `amount` フィールド = **保有量（最小単位）**。`bigint` で返ってくる。
- 表示時に `Number(...) / 1_000_000_000` して人間向けに変換。

## 5. トークンの送金（Transfer）

次は別人（受信者）へ 0.5 トークン送る。Solana では送金には **送信元の Token Account と受信者の Token Account の両方が必要**になる。受信者の ATA がまだなければ**先に作ってあげる**のが定石。

### 5.1 受信者の鍵ペアと ATA アドレスの導出

```typescript
const recipient = await generateKeyPairSigner();

const [recipientAssociatedTokenAddress] = await findAssociatedTokenPda({
  mint: mint.address,
  owner: recipient.address,
  tokenProgram: TOKEN_2022_PROGRAM_ADDRESS,
});

console.log(
  "\nRecipient Associated Token Account Address:",
  recipientAssociatedTokenAddress,
);
```

「誰に送るか」の宛先となる新しいウォレット鍵を作り、そのウォレットの ATA アドレスを `findAssociatedTokenPda` で導出する。**まだチェーン上に作られていなくてもアドレスだけは計算で出る**のが ATA / PDA の利点（4/28 5.1 で議論した通り）。

> **実務との対応**: 実務ではここで `generateKeyPairSigner()` ではなく「送り先のウォレットアドレス」（= 受信者から教えてもらった `Address` 文字列）を使う。今日は学習用に "送り先ウォレットも自分で作って" シミュレーションしている。重要なのは **送信者は受信者の ATA アドレスを `mint.address + 受信者のウォレットアドレス` だけで自動導出できる**こと。

### 5.2 受信者 ATA の作成

```typescript
const createRecipientAtaInstruction =
  await getCreateAssociatedTokenInstructionAsync({
    payer: feePayer,
    mint: mint.address,
    owner: recipient.address,
  });
```

4/28 5.2 で feePayer 自身の ATA を作ったのと**全く同じ命令**。違うのは `owner: recipient.address` の部分だけ。

> **誰が rent を払うのか**: `payer: feePayer` を渡しているので、**受信者の ATA を作るための rent も送信者の feePayer が払う**ことになる。実務では "送金者が事前に受信者 ATA を作ってあげる" のが慣習。受信者は SOL を持っていないかもしれないし、わざわざ受信者にトランザクションを発行させるのは UX が悪いため。
>
> なお ATA の rent も**閉じれば返ってくる**ので、実質的には「受信者がそのトークンを使い終わったら rent を回収できる」状態になる。

### 5.4 Transfer 命令の組み立て

```typescript
const transferInstruction = getTransferInstruction({
  source: associatedTokenAccountAddress,
  destination: recipientAssociatedTokenAddress,
  authority: feePayer.address,
  amount: 500_000_000n,
});
```

引数の意味:

- **`source`**: 送金元の Token Account / ATA アドレス（= feePayer の ATA）
- **`destination`**: 送金先の Token Account / ATA アドレス（= 受信者の ATA）
- **`authority`**: 送金を承認する権限者。**送金元 Token Account の `owner` フィールド** に書かれた人。今回は feePayer
- **`amount: 500_000_000n`**: 送金量。decimals=9 なので 0.5 トークン

> **`authority` は何者か**: ここで言う `authority` は **送金元 Token Account の `owner` フィールド** = **そのアカウントのトークンを使える人**を指す。4/28 4.4 で `getInitializeAccount2Instruction` の `owner` に `feePayer.address` を入れたのを思い出す。あの時に「保有者 = 操作権限者」として feePayer が登録されているので、ここの `authority` も feePayer になる。
>
> **「3つの owner 用語」のおさらい**: Solana では "owner" が3箇所で使われる紛らわしい構造になっている（4/20 4.6・4/28 1.5 を参照）:
> - **アカウントレベルの owner（システム）**: Token-2022 プログラム
> - **Token Account データ内の `owner` フィールド（アプリ）**: 保有者ウォレット = `feePayer.address`
> - **Transfer 命令の `authority` 引数**: 上の `owner` フィールドと一致する署名者
>
> 同じ概念が違う名前で出てくるだけで、構造的にはずっと一貫している。

## 6. トークンの凍結（Freeze）

凍結すると、その Token Account は**送金もバーンも一切できなくなる**。コンプライアンス対応や、犯罪利用が判明したアカウントを止める用途で使われる（USDC でも実際にこの仕組みが使われている）。

### 6.1 Freeze 命令の組み立て

```typescript
const freezeInstruction = getFreezeAccountInstruction({
  account: associatedTokenAccountAddress,
  mint: mint.address,
  owner: feePayer.address,
});
```

引数の意味:

- **`account`**: 凍結する Token Account のアドレス（= feePayer の ATA）
- **`mint`**: その Token Account が紐づいている Mint。Token-2022 が「この Mint の freezeAuthority と一致するか」を内部で照合するために必要
- **`owner`**: ここでは **Freeze Authority のアドレス**を指す。`Freeze` 命令の文脈では引数名は `owner` だが、実態は **Mint の freezeAuthority と一致しなければ実行が拒否される**

> **`owner` という名前のさらなる紛らわしさ**: ここの `owner` は「Token Account の保有者」ではなく「凍結権限者（= Mint の freezeAuthority）」のこと。**同じ "owner" という単語が、また別の意味で使われている**。前述の3種類に加えてここの4つ目は「実行に必要な権限の保有者」と読み替えるのが正解。

> **権限の出処**: 3 で `freezeAuthority: feePayer.address` を Mint 作成時に設定したので、feePayer が freezeAuthority として認められる。3 で `freezeAuthority` を渡していなかったら、ここで「Account does not support freezing」のようなエラーで弾かれる（4/20 で書いたコードのままなら凍結機能がそもそも有効になっていない）。

### 6.2 Freeze TX を送信

```typescript
const { value: freezeLatestBlockhash } = await rpc.getLatestBlockhash().send();

const freezeTxMessage = pipe(
  createTransactionMessage({ version: 0 }),
  (tx) => setTransactionMessageFeePayerSigner(feePayer, tx),
  (tx) =>
    setTransactionMessageLifetimeUsingBlockhash(freezeLatestBlockhash, tx),
  (tx) => appendTransactionMessageInstructions([freezeInstruction], tx),
```

## 8. トークンの焼却（Burn）

トークンを**永久に消す**操作。流通量（supply）が実際に減るのが特徴。

### 8.1 焼却前の残高確認

```typescript
const ataBeforeBurn = await fetchToken(rpc, associatedTokenAccountAddress, {
  commitment: "confirmed",
});

console.log(
  "\nAssociated Token Account Balance Before Burn:",
  Number(ataBeforeBurn.data.amount) / 1_000_000_000,
);
```

5/2 の流れだと feePayer の ATA は 1 から 0.5 を transfer したので、Burn 直前は 0.5 が残っている。

### 8.2 Burn 命令の組み立て（`Checked` 版）

```typescript
const burnInstruction = getBurnCheckedInstruction({
  account: associatedTokenAccountAddress,
  mint: mint.address,
  authority: feePayer.address,
  amount: 500_000_000n,
  decimals: 9,
});
```

引数の意味:

- **`account`**: 焼却対象の Token Account（= feePayer の ATA）
- **`mint`**: そのトークンの Mint アドレス
- **`authority`**: 焼却を承認する権限者 = **Token Account の owner フィールド**（= 保有者）。`Transfer` の `authority` と同じ意味
- **`amount: 500_000_000n`**: 焼却量。0.5 トークンを焼く（残高ぴったり）
- **`decimals: 9`**: **Mint の decimals** を呼び出し側からも明示。Token-2022 が「Mint の decimals と一致するか」を検証する

> **`Checked` 版の安全性**: 旧 `Burn` 命令は `decimals` 引数がなく、間違ったMintを burn してしまう事故があった（特に**動的にMintを決めるUI**では危険）。`BurnChecked` は呼び出し側に decimals を明示させ、Token-2022 が Mint と照合してから処理するので、誤Mint事故を防げる。**新規コードでは原則 `Checked` 版**を使う。同様の関係が `TransferChecked` と `MintToChecked` にもある。今日のスクリプトでは Transfer は `Checked` を付けない版を使ったが、本番では `TransferChecked` を選ぶのが望ましい。


```
Burn Transaction Signature: 4wDh1ziFeocRq8VypT9nWfne6thm79wfUWmgZVDPEsimrhSJsTVagQLpQJgZT8TUJd3dgRSX3fytDZvx8UGoLfhe
```

残高 0 になった。**Mint の supply も内部的に -0.5 されている**（このスクリプトでは確認していないが `fetchMint` で見れば確認可能）。

> **Burn と Transfer の本質的違い**: 両方とも送信者の残高は減るが、
> - **Transfer**: 受信者の残高が**同じだけ増える**。supply は変わらない
> - **Burn**: 受信者は存在せず**消滅する**。supply が**実際に減る**
>

## 9. アカウントのクローズ（Close Account）

残高ゼロになった Token Account を閉じて、**rent-exempt 用に預けてあった SOL を回収**する。これで Token Account の整理が完結する。

### 9.1 クローズの前提条件

- **残高がゼロ**（= 8 の Burn で 0 になっているのでOK）
- **`owner` フィールドの保有者が承認する**（または明示的に CloseAuthority が設定されていればそれ）
- **凍結されていないこと**（凍結中の口座は閉じられない。今日は 7 で Thaw 済みなのでOK）

### 9.2 Rent 回収先（destination）の準備


## 11. 操作と権限の対応関係（まとめ表）

今日触れた6つの操作は、**「権限を持つ人 → 命令 → Token-2022 が実行」** という構造で全部統一されている。違いは命令名と「誰の権限を要求するか」だけ:

| 操作 | 命令関数 | 必要な権限 | その権限はどこで決まるか |
|---|---|---|---|
| **発行** | `getMintToInstruction` | `mintAuthority` | Mint 作成時に設定 |
| **送金** | `getTransferInstruction` | 送金元 Token Account の `owner` | Token Account 作成時に設定 |
| **凍結** | `getFreezeAccountInstruction` | `freezeAuthority` | Mint 作成時に設定 |
| **解凍** | `getThawAccountInstruction` | `freezeAuthority`（同上） | Mint 作成時に設定 |
| **焼却** | `getBurnCheckedInstruction` | 対象 Token Account の `owner` | Token Account 作成時に設定 |
| **クローズ** | `getCloseAccountInstruction` | 対象 Token Account の `owner`（または `closeAuthority`） | Token Account 作成時に設定 |

**Mint レベルの権限**（mintAuthority / freezeAuthority）は **Mint 作成時に決める**。**Token Account レベルの権限**（owner / closeAuthority）は **Token Account 作成時に決める**。一度設定した authority は**所有者が `null` に変更（権限放棄）はできるが、`null` から復活はできない**点も共通。

## 12. 今回学んだキーコンセプト

1. **トークンの操作 = Token-2022 への命令送信**
   - 「Mint To」「Transfer」「Freeze」「Burn」「Close」など、**全部 Token-2022 が標準命令として用意している**
   - クライアント（TypeScript）は命令名と引数を組み立てて送るだけ。Solana では「トークンプログラムを自分で実装しない」というモデルが徹底している（4/20 9.7 で議論した Ethereum との違いそのもの）

2. **権限は事前に Mint / Token Account に焼き付ける**
   - mintAuthority / freezeAuthority は Mint 作成時に決め、後付けできない
   - 今日 freeze の前に **3 で `freezeAuthority` を Mint 作成命令に追記**したのはこのため
   - **「事後的に権限が必要になっても遡って付けられない」**ことを忘れがち

3. **`Checked` 版命令を選ぶのが新しい安全規範**
   - `BurnChecked` / `TransferChecked` / `MintToChecked` は decimals を一緒に渡し、Mint との不整合を Token-2022 がチェック
   - 動的にMintアドレスを扱うアプリでは事故防止に必須

4. **`fetchToken` で残高を読む**
   - 生の `getAccountInfo` + 自前パースではなく、`@solana-program/token-2022` の `fetchToken` を使えば `data.amount` で直接残高にアクセスできる
   - Mint 側を読みたい場合は `fetchMint` も用意されている

5. **rent-exempt の SOL は最後に Close で取り戻せる**
   - Token Account 1個あたり ≒ 0.00207 SOL
   - 残高 0 にしてから Close することで、destination で指定したアドレスに振り込まれる
   - 4/20 4.5 で議論した「SOL は預けてあるだけで戻ってくる」を、SOL の単位で実体験できた

6. **`pipe` + Factory + 型アサーションの3点セットは無限ループ**
   - 今日6つの操作を実装したが、骨格はすべて同じ
   - 「命令を1つ作る → blockhash 取る → pipe で組み立て → 署名 → 型アサーション → send」
   - **Solana 開発の体力は、このテンプレを何度書いても疲れない筋力**だと体感した

## 14. 全体の流れを図で整理

今日のスクリプト全体を1枚の図にまとめると（4/28 までの部分も含む）:

```
[feePayer 鍵ペア生成] → [1 SOL airdrop]
                              ↓
[mint 鍵ペア生成]
                              ↓
[Mint 作成 TX]   ※ freezeAuthority も指定（5/2 で追記）
                              ↓
[ATA 作成 TX]    ※ feePayer の ATA
                              ↓
─────────────────────── ★ ここから 5/2 ★ ───────────────────────
                              ↓
[MintTo TX] (1 トークン → feePayer の ATA)
                              ↓
[fetchToken] → balance: 1
                              ↓
[recipient 鍵ペア生成]
                              ↓
[recipient ATA 作成 TX]
                              ↓
[Transfer TX] (0.5 → feePayer から recipient へ)
                              ↓
[fetchToken x 2] → sender: 0.5, recipient: 0.5
                              ↓
[Freeze TX]  (feePayer の ATA を凍結)
                              ↓
[Transfer 試験 TX] → ★ "Account is frozen" で失敗（確認後コードは削除）★
                              ↓
[Thaw TX]    (feePayer の ATA を解凍)
                              ↓
[fetchToken] → balance: 0.5（焼却前）
                              ↓
[Burn TX]    (0.5 を焼却 = 残高 0)
                              ↓
[fetchToken] → balance: 0
                              ↓
[destination 鍵ペア生成]
                              ↓
[Close Account TX] → rent SOL が destination へ
                              ↓
[getBalance] → destination: 0.00207408 SOL
```

## 15. まとめ

今回の学習で、Solana 上の SPL トークン（Token-2022）の**実用的なライフサイクル全体**を実装した。

```
[Mint 設計図 (4/20)]
      ↓
[Token Account / ATA 保管庫 (4/28)]
      ↓
[発行 → 送金 → 凍結 → 解凍 → 焼却 → クローズ (今日)]
```

**3回をまたいで、SPL トークンを使う側が触る命令はほぼ全部体験できた**ことになる。残っているのは:

- **MintCloseAuthority 拡張**を使った Mint 自体のクローズ
- **Transfer Fee, Metadata, Permanent Delegate, Transfer Hook, Confidential Transfer** などの Token-2022 拡張機能
- 自作プログラム（Anchor / Pinocchio）から Token-2022 を **CPI 呼び出し**するパターン
