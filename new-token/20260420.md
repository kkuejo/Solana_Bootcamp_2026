# @solana/kit で新規トークン（Mint Account）を作る

**日付**: 2026年4月20日
**学習内容**: Solana公式の日本語Bootcamp動画の続きを視聴し、**Token-2022プログラム**を使って新しいトークン（Mint Account）をLocalnet上に発行するTypeScriptスクリプトを書いた。ライブラリは最新の `@solana/kit`（旧 `@solana/web3.js` の後継）を使用。Surfpoolをローカルチェーンとして起動し、`npx tsx` で実行。

## 0. 本日の位置づけ

前回（4/11）に宣言した通り、今回から本格的に **Solana公式の日本語Bootcamp動画** を軸に学習を進めている。本日視聴したのは前回の続きの回で、テーマは「Solana上で自分のトークンを発行する」。

Solanaでは、USDCのようなステーブルコイン、NFT、ゲーム内通貨、ポイントなど、**あらゆる独自トークンは同じ仕組み**で発行できる。そのベースとなるのが **Mint Account（ミントアカウント）** である。本日はこの Mint Account を作るスクリプトを、ゼロから書いた。

## 1. 学習の全体像

今日書いたスクリプト（`new-token.ts`）が最終的にやることは、たった一つ:

> **「Localnetチェーン上に、自分だけが鋳造権（Mint Authority）を持つ新しいトークンの発行元アカウントを作成する」**

これだけだが、裏側では以下の処理が行われている:

1. Localnetに接続する（RPC / WebSocket）
2. 手数料を払う用のキーペア（`feePayer`）を生成し、1 SOL airdropする
3. Mint Account 用のキーペアを生成する
4. Mint Accountに必要なディスクサイズを取得する
5. そのサイズに対する **Rent-exempt（レント免除）** に必要なSOL量を問い合わせる
6. システムプログラムで **空のアカウント** を作成する命令を組み立てる
7. Token-2022プログラムで そのアカウントを **Mintとして初期化** する命令を組み立てる
8. 上記2つの命令を1つのトランザクションにまとめる
9. トランザクションに署名し、チェーンに送信する
10. 成功したらMintアドレスと署名（Signature）を表示する

前回（4/11）はテンプレートが全部やってくれていた部分を、**今回は自分の手で1行ずつ書く**のが大きな違いである。

## 2. プロジェクトのセットアップ

### 2.1 プロジェクトディレクトリと package.json

```bash
mkdir new-token
cd new-token
npm init -y
```

`npm init -y` は、質問に全て「はい（デフォルト）」で答えて `package.json` を自動生成するコマンド。`-y` を付けないと対話モードになる。

### 2.2 必要なライブラリのインストール

```bash
npm i @solana/kit @solana-program/system @solana-program/token-2022
```

3つのパッケージをインストールする:

- **`@solana/kit`**: 旧 `@solana/web3.js` の**次世代版**。Solanaが公式に推奨する新しいクライアントライブラリで、Tree-shaking対応（使っていない機能はバンドルされない）・型安全性の強化・関数型スタイルが特徴。
- **`@solana-program/system`**: Solanaの**システムプログラム**（アカウントの作成や SOL 転送を担う、最も基礎的なビルトインプログラム）を呼び出すためのラッパー。
- **`@solana-program/token-2022`**: **Token-2022プログラム**（従来のSPL Tokenの後継で、手数料徴収や金利付きトークンなどの拡張機能をサポート）を呼び出すためのラッパー。

> **補足**: Solana界隈では2024年頃から、ライブラリ名が `@solana/web3.js` → `@solana/kit` に置き換わり、プログラムごとのクライアントも `@solana/spl-token` → `@solana-program/token-2022` のような命名規則に統一されつつある。動画も最新のこの命名に従っている。

### 2.3 スクリプトファイルの作成

```bash
touch new-token.ts
```

空のTypeScriptファイルを作成する。ここに今日のロジックを書いていく。

### 2.4 TypeScript実行環境のインストール

```bash
npm i -D tsx
```

`tsx` は「TypeScriptファイルをコンパイルせずに**そのまま実行できる**」ツール。

コマンドの各部分の意味:

- **`npm`**: Node.jsのパッケージマネージャ
- **`i`**: `install` の短縮形。`npm install` と全く同じ意味で、パッケージをダウンロードして `node_modules/` 配下に配置し、`package.json` の依存リストに追記する。2.2 で使った `npm i @solana/kit ...` の `i` もこれ。タイプ数を減らすための慣例的な省略表記で、多くの npm コマンドに同様の短縮形（`rm` = `uninstall`, `ci` = `clean-install` など）がある。
- **`-D`**: `--save-dev` の短縮形。**開発用依存（`devDependencies`）**としてインストールするオプション。本番環境には不要だがローカル開発では使う、というライブラリに付ける。対して `-D` を付けない通常の `npm i xxx` は `dependencies` に入り、本番環境にもインストールされる。
- **`tsx`**: インストール対象のパッケージ名

> **補足**: `tsx` を `-D`（dev依存）にしている理由は、「このスクリプトを**実行する**ためだけに使うツール」であって、**完成した成果物には含まれない**から。開発中は `npx tsx new-token.ts` のように `.ts` を直接動かせるが、本番では通常 `tsc` などで `.ts` を `.js` にビルドし、`node dist/new-token.js` のように `.js` を実行する。ここでいう「ランタイム」は「プログラムが実際に動いている実行環境」のこと。本番環境で実際に `require` / `import` されるのはビルド後のアプリ本体とその依存（例: `@solana/kit`）であり、`tsx` は本番ランタイムでは使わず、開発・検証フェーズで使う。つまり `tsx` がなくても本番の `.js` は動くため、`tsx` はランタイム依存ではなく `dependencies` ではなく `devDependencies`（`-D`）に入れるのが適切。一方で `@solana/kit` などは本番でも必須なので通常の `dependencies` に入れる。

### 2.5 package.json を ESM に変更

`package.json` の `"type": "commonjs"` を `"type": "module"` に書き換える。

```json
{
  "type": "module",
  ...
}
```

これは**地味だが重要な設定**である。なぜ必要なのかを順を追って説明する。

#### Node.jsには2つのモジュールシステムがある

Node.jsには歴史的に**2つの異なるモジュール形式**が並存している:

| 形式 | 呼び方 | インポート構文 | エクスポート構文 | 歴史 |
|---|---|---|---|---|
| **CommonJS (CJS)** | `commonjs` | `const x = require("x")` | `module.exports = ...` | Node.js独自の古い仕組み（2009〜） |
| **ECMAScript Modules (ESM)** | `module` | `import x from "x"` | `export ...` | JavaScript言語標準の新しい仕組み（2015〜、Node.js対応は2019〜） |

どちらを使うかは、**プロジェクトを "どっち側の世界として動かすか"** という問題で、ファイルごとに切り替える（`.mjs` / `.cjs` 拡張子）か、プロジェクト単位で `package.json` の `"type"` フィールドで宣言する。

- `"type": "commonjs"`（または未指定）: `.js` / `.ts` ファイルは **CommonJS** として解釈される
- `"type": "module"`: `.js` / `.ts` ファイルは **ESM** として解釈される

#### 何が困るのか — `@solana/kit` は ESM 専用パッケージ

`@solana/kit` は **ESMでしか配布されていない**（CommonJS版は同梱されていない）。つまり中身が `export const foo = ...` のような ESM 形式で書かれており、`require()` で読み込もうとすると Node.js は解釈に失敗する。

新しめのSolanaエコシステムのパッケージ（`@solana/kit`, `@solana-program/*` など）は揃って ESM-only になっている。これはESMがJavaScriptの**言語標準**であり、Tree-shaking（使っていないコードをビルド時に落とす最適化）もESMのほうが効きやすい、という理由から業界全体のトレンドになっている。

#### `"type": "commonjs"` のままだと具体的に何が起こるか

`npm init -y` は `"type"` を明示しない（= CommonJSがデフォルト）`package.json` を生成する。この状態で `new-token.ts` を `tsx` で実行しようとすると、tsx は「CommonJSモードだな」と判断し、ファイル冒頭の:

```typescript
import { airdropFactory, ... } from "@solana/kit";
```

を内部的に:

```javascript
const { airdropFactory, ... } = require("@solana/kit");
```

相当の処理に変換しようとする。しかし `@solana/kit` はCommonJS形式で `require` されることを想定していないため、起動時に**以下のようなエラーで落ちる**:

```
Error [ERR_REQUIRE_ESM]: require() of ES Module /path/to/@solana/kit/...
is not supported. Instead change the require() to a dynamic import() ...
```

もしくは、Node.jsのバージョンやtsxの設定によっては:

```
SyntaxError: Cannot use import statement outside a module
```

のようなエラーになる。どちらも原因は同じで「**ESMのパッケージをCommonJS環境から読もうとした**」という食い違い。

#### 解決策: `"type": "module"` に変更する

`package.json` の `"type"` を `"module"` にすることで、プロジェクト全体を**ESMの世界**として宣言する。するとtsxは `import` 文をそのまま ESM として扱い、`@solana/kit` を正しく読み込める。

つまりこの1行の書き換えは、**「このプロジェクトは新しい方のJavaScript標準で動かしますよ」と宣言する**ことに他ならない。`npm init -y` のデフォルトが古い方（CommonJS）なので、最新パッケージを使うときに明示的に切り替える必要がある、というのが現状のNode.js事情である。

> **補足**: ESM化すると `require("xxx")` が使えなくなる・`__dirname` / `__filename` が使えなくなる、といった副作用もあるが、新規プロジェクトで `@solana/kit` 系を使う限り全て `import`/`import.meta.url` で代替できるので実害はない。

## 3. Surfpool でローカルチェーンを起動

```bash
surfpool start
```

**Surfpool** は、Solanaの `solana-test-validator` と役割は同じ「ローカルでSolanaチェーンを立ち上げるツール」だが、以下のメリットがある:

- 起動が速い（`solana-test-validator` より軽量）
- Mainnetの状態をフォークして試せる
- ダッシュボードUI付き

今回は単に「Localnetとして使う」だけなので、起動しっぱなしにしておけばOK。デフォルトで以下のポートが開く:

- **RPC（HTTP）**: `http://localhost:8899`
- **WebSocket**: `ws://localhost:8900`

スクリプト側はこの2つのエンドポイントに接続する。

### なぜ2種類のエンドポイントが必要なのか

この2つは**通信モデルが全く違う**ため、用途も明確に分かれている。

#### RPC（HTTP, port 8899）: 一問一答の問い合わせ・命令送信

**リクエスト→レスポンス**型の通常のHTTP通信。クライアントから1回リクエストを投げて、1回レスポンスが返ってきて終わり。使う場面:

| 用途 | 対応するRPCメソッド | 今日のスクリプトで使った箇所 |
|---|---|---|
| アカウントの情報を取得 | `getAccountInfo` | — |
| 残高を取得 | `getBalance` | — |
| Rent免除に必要なSOLを問い合わせ | `getMinimumBalanceForRentExemption` | **4.5** |
| 最新のブロックハッシュを取得 | `getLatestBlockhash` | **4.9** |
| トランザクションを送信 | `sendTransaction` | 4.13（内部で使用） |
| airdropを要求 | `requestAirdrop` | 4.3（内部で使用） |
| プログラムから派生するアカウント一覧を取得 | `getProgramAccounts` | — |

つまり「**チェーンに何か問い合わせる**」「**チェーンに命令を送りつける**」という一方向的な操作は全てHTTP RPCで行う。`.send()` を最後に付けるのがこのパターン。

#### WebSocket（port 8900）: 変化を"待つ"ための待ち受けチャネル

一度接続すると**開きっぱなし**になり、サーバー側から「何か起きたよ」とプッシュ通知が届く仕組み。クライアントは「このアカウントが変わったら教えて」「このTXが確定したら教えて」と**待ち受け登録（subscribe）**を行う。使う場面:

| 用途 | 対応する待ち受けメソッド | 今日のスクリプトで使った箇所 |
|---|---|---|
| トランザクション確定を待つ | `signatureSubscribe` | **4.3**（airdrop確定待ち）、**4.13**（TX送信後の "confirmed" 待ち） |
| アカウントの変化を監視 | `accountSubscribe` | — |
| プログラム関連アカウントの変化を監視 | `programSubscribe` | — |
| 新しいブロック（スロット）を監視 | `slotSubscribe` | — |
| プログラムのログをリアルタイムで見る | `logsSubscribe` | — |

**何かが終わるまで待つ**、**リアルタイムで変化を追う**、といった用途は全てWebSocket側。HTTP RPC のような `.send()` は基本的に使わず、待ち受けを開始して通知を受け取る（必要なら待ち受け解除する）という流れで扱う。

#### 「送る」と「待つ」の両方が必要な処理

`airdropFactory` や `sendAndConfirmTransactionFactory` のように、**両方のクライアントを受け取る関数**があるのは、裏で以下のコンビネーションを行うため:

```
sendAndConfirmTransaction の内部動作:
  1. [HTTP] sendTransaction でトランザクションを送信
  2. [WS]   signatureSubscribe でそのTX署名の確定通知を待ち受け
  3. [WS]   "confirmed" の通知を受信
  4. Promise を resolve（呼び出し側に制御を返す）
```

HTTP だけだと「送った直後に一瞬で返ってきてしまい、本当に確定したか分からない」、WebSocket だけだと「そもそも送れない」ので、**両方揃って初めて"確定まで待って成功を返す"処理が書ける**。

ここでの **Promise** は「あとで結果が返ってくる非同期処理の入れ物」を意味する。`sendAndConfirmTransactionFactory(...)(...)` は、送信と確定待ちが終わるまで `pending`（処理中）の Promise を返し、"confirmed" を受け取った時点で `resolve`（成功確定）される。`resolve` されると、呼び出し側の `await` が解除されて次の行へ進む。逆に失敗時は `reject` され、`try/catch` の `catch` 側で扱う。

今回の `new-token.ts` では WebSocket メソッドを直接呼ばず、`airdropFactory` / `sendAndConfirmTransactionFactory` に `rpcSubscriptions` を渡すことで、ライブラリ内部で `signatureSubscribe` 相当の待ち受けが実行される。

#### ポート番号の覚え方

Solanaでは慣習的に:

- **RPC（HTTP） = 8899**（Localnet）、mainnetなら `https://api.mainnet-beta.solana.com`
- **WebSocket = 8900**（Localnet）、mainnetなら `wss://api.mainnet-beta.solana.com`

Localnetでは**HTTP のポート +1 がWebSocket**、と覚えるとよい。どちらも Surfpool / solana-test-validator が同時に立ち上げてくれる。

#### フロントエンドから見ると？

前回（4/11）のようなブラウザUI付きDAppの場合、ウォレット操作（署名承認）やカウンター値の更新表示のために**フロントエンドからも両方に接続**する。たとえば:

- HTTP: ユーザーが「Increment」ボタンを押したら送信
- WebSocket: 他人が同じカウンターをIncrementしたら画面を自動更新

という使い分けになる。今日のようなCLIスクリプトでも、ブラウザDAppでも、**基本構造は同じ2チャネル**ということである。

### 4.1 インポート

いきなり大量のインポートだが、**1つ1つが後で必ず1回使われる関数**である。ざっくりグルーピングすると:

| グループ | 関数 | 役割 |
|---|---|---|
| 接続 | `createSolanaRpc`, `createSolanaRpcSubscriptions` | RPC/WSのクライアントを作る |
| キーペア | `generateKeyPairSigner` | 署名できる鍵を生成 |
| SOL補充 | `airdropFactory`, `lamports` | airdropする・単位変換 |

| 署名・送信 | `signTransactionMessageWithSigners`, `sendAndConfirmTransactionFactory`, `getSignatureFromTransaction` | 署名してチェーンに送る |
| 命令（System） | `getCreateAccountInstruction` | 空アカウント作成命令 |
| 命令（Token-2022） | `getInitializeMintInstruction`, `getMintSize`, `TOKEN_2022_PROGRAM_ADDRESS` | Mint初期化命令 |

`@solana/kit` の流儀は **「細かい関数を `pipe` で合成して組み立てる」** 関数型スタイル。旧 `@solana/web3.js` の「クラスのメソッドを呼ぶ」形とは大きく異なる。

### 4.2 RPC / WebSocket クライアントの作成

```typescript
const rpc = createSolanaRpc("http://localhost:8899");
const rpcSubscriptions = createSolanaRpcSubscriptions("ws://localhost:8900");
```

- **`rpc`**: 「チェーンに問い合わせる」「命令を送る」の**リクエスト/レスポンス型**通信（HTTP）。
- **`rpcSubscriptions`**: 「チェーン上の変化を待ち受ける」（例: トランザクションが承認されたら教えて）**プッシュ通知型**通信（WebSocket）。

`airdrop` や `sendAndConfirmTransaction` のような「成功まで待つ」系の関数は、裏でWebSocketを使って結果をリアルタイムに受け取るため、両方必要になる。

### 4.3 手数料支払い用キーペアの生成と Airdrop

```typescript
const feePayer = await generateKeyPairSigner();

await airdropFactory({ rpc, rpcSubscriptions })({
  recipientAddress: feePayer.address,
  lamports: lamports(1_000_000_000n),
  commitment: "confirmed",
});
```

Solanaでは**どんなトランザクションにも手数料（fee）が必要**。その手数料を払うアカウント（= `feePayer`）を用意する必要がある。

- **`generateKeyPairSigner()`**: 新しい鍵ペア（秘密鍵＋公開鍵）を生成し、署名機能付きのオブジェクトを返す。`.address` でPublicKey（アドレス）を取り出せる。秘密鍵は通常の使い方では `Uint8Array` として直接取り出さず、署名処理の内部で使われる（= 秘密鍵を露出させない設計）。どうしても保存・書き出しが必要な場合は `generateKeyPairSigner(true)` のように extractable で生成し、`writeKeyPairSigner(...)` などで明示的にエクスポートする。
- **`airdropFactory`**: これは`@solana/kit` が**最初から提供している関数**（4.1 のインポート文で読み込んでいるもの）。自作したものではない。名前に `Factory` と付いている通り、**「airdrop関数を作って返す」関数**（= 高階関数、つまり**関数を返す関数**）として設計されている。
- **`airdropFactory({ rpc, rpcSubscriptions })(...)` という2段階呼び出し**: これは `@solana/kit` 全体に共通する**Factoryパターン**の書き方で、次のような2ステップになっている:
  1. `airdropFactory({ rpc, rpcSubscriptions })` を呼ぶ → **rpc/ws接続を内部に埋め込んだ airdrop 関数** が返ってくる
  2. 返ってきた関数に `({ recipientAddress, lamports, commitment })` を渡す → 実際に airdrop が実行される
- **なぜわざわざ2段階なのか**: 「同じ rpc 設定で何度も airdrop したい」場面で無駄がないから。1段階目の戻り値を変数に保存しておけば、以降は rpc/ws の指定を毎回書かずに済む:
  ```typescript
  const airdrop = airdropFactory({ rpc, rpcSubscriptions });  // 関数を作って使い回す
  await airdrop({ recipientAddress: alice.address, lamports: lamports(1_000_000_000n), commitment: "confirmed" });
  await airdrop({ recipientAddress: bob.address,   lamports: lamports(2_000_000_000n), commitment: "confirmed" });
  ```
  今回のスクリプトでは1回しか使わないので、わざわざ変数に取らず作って即呼んでいる。同じパターンは後ほど出てくる `sendAndConfirmTransactionFactory` でも使われる。
- **`lamports(1_000_000_000n)`**: **1 SOL = 10^9 lamports**。つまり `1_000_000_000n` は 1 SOL を意味する。`_` は桁を読みやすくするための区切りで、`1000000000n` と値は同じ。末尾の `n` は `BigInt`（64ビット整数）リテラル。SOL量は `bigint` で扱うのが `@solana/kit` の流儀。
- **`commitment: "confirmed"`**: 「過半数のバリデーターに承認された段階で成功とみなす」設定。`processed`（送ったら即返す）より厳密、`finalized`（完全確定まで待つ）より速い、バランスの取れた選択肢。

これで feePayer のアドレスに 1 SOL が入る。

### 4.4 Mint用キーペアの生成

```typescript
const mint = await generateKeyPairSigner();
```

**Mint Account 自身のアドレス**用のキーペアを生成する。Solanaでは「新しいアカウント = 新しいアドレス」なので、Mintアカウントを作る前にそのアドレスを決める必要がある。

ここでは「ランダムな鍵ペアをMintアドレスとして使う」方針を取っている（PDAではなく、通常のKeypair方式）。
通常のKeypairは秘密鍵を持つため、Phantom などのウォレットで管理・署名に利用できる（EOAに近い）。一方PDAは秘密鍵を持たず、役割はEthereumのCAに近いが、CAのようにそのアドレスにコードが載るわけではなく、プログラム側のロジックで制御される。

### 4.5 Mintアカウントに必要なサイズと Rent の計算

```typescript
const space = BigInt(getMintSize());

const rent = await rpc.getMinimumBalanceForRentExemption(space).send();
```

- **`getMintSize()`**: Token-2022 のMintアカウントが取るべき**データサイズ（バイト数）**を返す関数。Mintの内部フィールド（decimals, mintAuthority, supply など）の合計値。拡張なしの素のMintでは、ライブラリ内で定義された固定レイアウトに基づく実質固定値（= コード上で決まっている値）を返す。拡張機能を足す場合はその分だけ必要サイズが増えるため、返る値も変わる。
- **`BigInt(...)`**: 数値を `bigint` 型に変換。下のRPC呼び出しが `bigint` を要求するため。
- **`rpc.getMinimumBalanceForRentExemption(space)`**: **そのサイズのアカウントを「レント免除」にするために必要な最小SOL量**をRPCに問い合わせる。
- **`.send()`**: `@solana/kit` では、RPC呼び出しは「クエリを作る」→「`.send()` で実行」という2段階になっている。遅延実行（Lazy）スタイル。

> **Rent（レント）とは**: Solanaの全アカウントは、チェーン上にデータを保持している限り**「レント（家賃）」**を払う必要がある。ただし、**そのアカウントに一定額以上のSOL**を入れておけば永久にレント免除（Rent-exempt）になる。その最小残高がこの値。Mintを永続的に使いたいので、必ずrent-exemptな額を入れる。

#### 補足: その rent-exempt の SOL は "どのアドレス" が保持するのか？

一瞬混乱しやすいポイントなので明確にしておく:

> **答え**: **Mint Account自身のアドレス（`mint.address`）がそのSOLを保持する。**

**Token-2022のプログラムアドレスではない**し、**feePayerのアドレスでもない**。

これを理解するには、Solanaの「アカウント」の性質を押さえておく必要がある:

- Solanaでは**全てのアカウントが個別に lamports 残高を持っている**。ユーザーのウォレットだけでなく、Mint Account も、後で作る Token Account も、データを格納するためだけのアカウントも、**全員が自分の財布を持っている**。
- 「**owner**（所有プログラム、= `programAddress: TOKEN_2022_PROGRAM_ADDRESS`）」と「**lamportsを保持するアドレス**」は**別の概念**。
  - **owner** は「このアカウントの中身のデータを書き換える権限を持つプログラム」のこと
  - **lamports** は単にそのアカウント自身の残高

今日のスクリプトで言うと:

```
feePayer (1 SOL持つ)
   │
   │ CreateAccount命令で rent 分を送金
   ▼
Mint Account (mint.address)
  ├─ lamports: rent （← ここがrent-exemptのSOL保有先）
  ├─ owner: TOKEN_2022_PROGRAM_ADDRESS
  └─ data: [decimals, mintAuthority, supply, ...]
```

`getCreateAccountInstruction` の引数 `lamports: rent` は「**新しく作るMint Accountの初期残高**」を指定している。つまり feePayer から rent 分のSOLが Mint Account に移動し、それ以降はMint Accountの中に入りっぱなしになっている。

#### Q: `owner: TOKEN_2022_PROGRAM_ADDRESS` ってどういう意味？ mint.address が自分のオーナーなのでは？

これはSolanaで**最も混乱しやすいポイント**なので、単独で節を設けて説明する。

##### 答え: Solanaの「owner」は「所有者」ではなく「書き換え権限を持つプログラム」のこと

直感的には「Mint Account の"オーナー"と言えば、そのMintのアドレス自身、あるいは作成者（feePayer）でしょ？」と思うはずである。しかしSolanaで「**owner**」という言葉は**普段の日本語の"所有者"とはズレた意味**で使われている。

Solanaにおける `owner` の定義:

> **owner = そのアカウントの中身のデータを書き換える権限を持つ"プログラム"のアドレス**

つまり `owner` は**必ずプログラムのアドレス**であり、**人間のウォレットでもなく、アカウント自身でもない**。

##### なぜMintのownerがToken-2022なのか

Mint Accountの中には `decimals`、`mintAuthority`、`supply`（発行済み総量）などのデータが格納されている。このデータを**解釈し、書き換えるロジック**を持っているのは Token-2022 プログラムだけ。例えば:

- 「新しくトークンをmintしたから supply を +100 してほしい」→ Token-2022 に命令を送る → Token-2022 が Mint Account の supply を書き換える
- 「Mint Authorityを別の人に変えたい」→ Token-2022 に命令を送る → Token-2022 が Mint Account の mintAuthority フィールドを書き換える

**Solanaのランタイムは、`owner` に指定されたプログラム以外がそのアカウントのデータやlamportsを勝手に書き換えることを物理的に禁止している**。これがowner制度の核心である。

つまり `owner: TOKEN_2022_PROGRAM_ADDRESS` は「**このMint Accountの中身を書き換えていいのは Token-2022 だけですよ**」という**立入権限の宣言**であって、「Token-2022 がこのMintを所有している」という意味ではない。

##### 日常語に翻訳すると

英語の "owner" を「管轄プログラム」「書き換え担当」と読み替えると納得しやすい:

| 原文 | より正確な日本語訳 |
|---|---|
| `owner: TOKEN_2022_PROGRAM_ADDRESS` | **「書き換え担当プログラム: Token-2022」** |
| 「Mint Account is owned by Token-2022」 | 「Mint Account は Token-2022 の管轄下にある」 |

##### "所有"ではなく"管轄"と考える比喩

比喩で言えば:

- **Mint Account = 銀行の口座データ**
- **owner = TOKEN_2022_PROGRAM_ADDRESS = その口座を管理している銀行システム**
- **mintAuthority = feePayer = その口座の名義人（操作を依頼できる人）**

銀行口座の「**所有者**」と聞いて名義人（= 名義人がfeePayerに相当）を思い浮かべるのが普通だが、Solanaの `owner` は**銀行システム（= Token-2022）**のほうを指している。名義人に相当するのはむしろ `mintAuthority` フィールドのほう。

##### 「操作を依頼できる人」と「データを書き換える権限」の2層構造

整理するとMintには**2種類の"権限を持つ人"**が関わっている:

| 項目 | 誰が | 何ができる |
|---|---|---|
| **owner**（アカウント単位のプロパティ） | **Token-2022 プログラム** | Mintのデータ（decimals, supply等）を**実際に書き換える権限** |
| **mintAuthority**（データ内のフィールド） | **feePayer のアドレス** | Token-2022 に「追加発行してくれ」と**命令を出す権限** |

この2つは全く別のレイヤー。`owner` が**システムレベルの実行権限**、`mintAuthority` が**アプリケーションレベルの操作権限**である。

- 「追加発行したい人」→ mintAuthority（= feePayer）が Token-2022 に命令を送る
- 「その命令を受けて実際にMint.supplyを+100する処理」→ Token-2022が実行する（ownerだから書き換えられる）
- 「mintAuthorityじゃない他人が勝手に命令を送ってきたら？」→ Token-2022 が内部チェックで拒否する
- 「Token-2022 以外のプログラムが勝手にMint.supplyを書き換えようとしたら？」→ Solanaランタイムが物理的に拒否する

##### Token Account でも同じ構造

次回作るToken Accountでも同じパターンが現れる:

- Token Accountの `owner` = **Token-2022**（システムレベル）
- Token Accountの中のデータに **`owner` という名前のフィールド**がある（※これがまた紛らわしい！）= **そのToken Accountの残高を使える人**（アプリケーションレベル）

つまり Solana の SPL トークンの世界では、**"owner" という単語が2箇所で違う意味で使われる**。

- **アカウントのowner**（システムレベル）: 必ず Token-2022
- **Token Accountデータ内のownerフィールド**（アプリレベル）: トークン保有者のウォレット

名付けが悪いだけで、概念は明確に分かれている。**「システム側のownerは必ずプログラム」「データ内のownerはユーザーのウォレット」**と覚えておけばOK。

##### 結論

- **Mint Account の `owner` は TOKEN_2022_PROGRAM_ADDRESS** で正しい
- これは「Token-2022がMintの管轄プログラム（= 書き換え権限を持つ唯一の存在）」であるという意味
- 「mint.address が自分のオーナー」ではないし、「feePayer がオーナー」でもない
- feePayer は `mintAuthority` として、Token-2022 に**命令を出せる立場**として別に記録されている

Ethereumの感覚で言うと、ERC20の`balanceOf`テーブルを書き換える権限はERC20コントラクト自身しか持たない、というのと同じ。Solanaではそれが**アカウントのメタデータ（owner）として明示的に記録されている**のが違いと言える。

#### Ethereumとの違い

Ethereumだと「ETHを保持できるのはEOA（ユーザーアドレス）とスマートコントラクトアドレスだけ」で、ERC20の**残高データ**はコントラクトの `mapping` に記録される。

Solanaでは逆で、**どんな用途のアカウントでもSOLを持てる**。Mint Accountはトークンの"設定情報"を保持するためのアカウントだが、それ自身も自分の財布（= lamports残高）を持っている。この財布が rent-exempt の要件を満たしていれば、そのアカウントは消えずに残り続ける。

#### SOLはいずれ誰の手に戻るのか

将来そのMint Accountを**閉じる**（`CloseAccount` 命令を使う）と、保持されていたSOLが指定した回収先アドレスに送り返される。つまり rent-exempt のSOLは**「アカウントをチェーン上に存在させ続けるための保証金」**であり、消費されるわけではない。アカウントを閉じれば戻ってくる。なお Token-2022 で Mint を閉じるには、`MintCloseAuthority` 拡張を有効化したうえで、通常は供給量（`supply`）を 0 にするなどの条件を満たす必要がある。今回のコードはこの拡張を使っていないため、Mintを閉じて rent を回収するフローは実装していない。

> **補足**: Mint Account の場合、現実的には「発行済みトークンが流通している状態で閉じる」ことは稀なので保証金は据え置きになりやすい。一方、次回以降学ぶ **Close Token Account**（残高ゼロのToken Accountを閉じる操作）では、このSOLの回収が実際に発生する。

#### Q: `feePayer` は「デプロイした人のウォレット」のイメージ？ `mint.address` は Ethereum の CA に該当する？

**結論から言うと、両方ともおおむねイメージ通りで正しい。ただし細部に Solana 特有のひねりがある**ので、Ethereum の EOA / CA の概念と対応づけて整理する。

##### feePayer ≒ Ethereum の "デプロイ者のEOA"

今日の役割から見ると:

- トランザクション手数料を払う
- 新しいMint作成命令を発行する
- 作成される Mint の**Mint Authority**（鋳造権限の持ち主）に指定されている

これは Ethereum で言えば**「ERC20をデプロイしてOwner権限を持つEOA」**に完全に対応する。Ethereumなら `msg.sender` として `ownable` コントラクトの `owner` になる人、Solanaなら `mintAuthority` として設定される人 — どちらも**人間が秘密鍵で管理するウォレット**である点で同じ。

> **注意**: 今日のスクリプトでは簡単のため `feePayer` を `generateKeyPairSigner()` で**その場で生成した使い捨てキーペア**にしている。実務では Phantom などの**開発者の実ウォレット**をここに使う。つまり今日のコードは "コードで動作確認するための仮想の人" を作っているだけで、本質的な役割としては「開発者のウォレット = EOA」で正しい。

##### mint.address ≒ Ethereum の CA（コントラクトアドレス）— ただし実装は違う

Mint Account は:

- **人間のウォレットではなく、データを保持するためのアカウント**（残高、decimals、mintAuthority、supply などを格納）
- **書き換えられるのはToken-2022プログラムだけ**（`owner = TOKEN_2022_PROGRAM_ADDRESS` のため）
- 外部（= TypeScriptクライアント）からは**Token-2022プログラムへの命令経由でしか操作できない**

という性質から、**立ち位置はEthereumのCAとほぼ同じ**。つまり「トークンの実体・ロジックを司るアドレス」= Ethereum なら `USDC.sol` のコントラクトアドレス、Solana なら `mint.address` に相当する。

##### ERC20 のように、トークン自体に独自関数を足せるのか？

ここで Ethereum の感覚だと、「ERC20 をベースに `transfer()` や `approve()` 以外の独自関数をトークンに組み込めるのでは？」と考えたくなる。

Ethereum の ERC20 では、トークンごとにコントラクトをデプロイする:

```
USDC ERC20 コントラクト
  ├─ transfer()
  ├─ approve()
  ├─ mint()
  ├─ burn()
  └─ 自作関数()
```

そのため、ERC20 をベースにしつつ:

```solidity
function claimReward() ...
function lockTokens() ...
function specialTransfer() ...
```

のような独自関数を、同じトークンコントラクトの中に追加できる。

一方、Solana の Token-2022 では構造が違う。

```
Token-2022 Program（共通）
  ├─ transfer
  ├─ mintTo
  ├─ burn
  ├─ freeze
  └─ Token-2022 の標準拡張

あなたの Mint Account
  └─ このトークンの設定・供給量・権限など

あなたの Token Account / ATA
  └─ 各ユーザーの残高
```

つまり、自分のトークンごとに新しい Token Program を作っているわけではない。**共通の Token-2022 Program を使い、その上に自分の Mint Account を作っている**だけである。

そのため、Mint Account に:

```
myCustomFunction()
```

のような関数を直接生やすことは基本できない。`mint.address` は Ethereum の CA に近い立ち位置ではあるが、**コードが載っているコントラクトそのものではなく、Token-2022 Program が読むデータアカウント**である。

Solana で独自ロジックを入れたい場合は、通常は **別の自作プログラム**を作る。

```
あなたの自作プログラム
  ├─ claimReward()
  ├─ lockTokens()
  ├─ specialTransfer()
  └─ 内部で Token-2022 Program を呼び出す
```

この自作プログラムが、必要に応じて Token-2022 Program に対して CPI（Cross Program Invocation）で:

```
mintTo
transfer
burn
```

などを実行する。

つまり設計思想としては:

```
Ethereum:
トークンコントラクト自体に独自関数を追加できる

Solana:
Token-2022 自体には自由に関数追加できない
独自ロジックは別の自作プログラムに書く
そのプログラムから Token-2022 を呼び出してトークンを動かす
```

となる。

例外的に、Token-2022 には標準の **Extension** が用意されている。例えば Transfer Fee、Metadata、Permanent Delegate、Transfer Hook、Confidential Transfer などである。特に **Transfer Hook** を使うと、転送時に自作プログラムを呼ばせることができる。

ただしこれも「Mint に自由な関数を追加する」というより、**Token-2022 が用意した拡張ポイントに自作プログラムを接続する**イメージである。

まとめると、Solana では **「トークン = 関数を持つコントラクト」ではなく、「Mint / Token Account = データ、自作プログラム = ロジック」** と分けて考えるのが近い。

しかし実装レベルで見ると2つ重要な違いがある:

**違い①: Solanaの通常アカウントはキーペアから生まれる**

- **Ethereum CA**: 秘密鍵が**存在しない**。デプロイ時に deployer address + nonce から自動計算される（または CREATE2）。
- **Solana Mint Account（今日の方式）**: `generateKeyPairSigner()` で**秘密鍵つきで生成**される。作成時に "本人同意の署名" として秘密鍵が使われる。

ただし、**作成後はこの秘密鍵はほぼ使い道がない**。Mintのデータを書き換えられるのはToken-2022プログラムだけなので、Mint の秘密鍵を持っていても「追加発行」も「decimalsの変更」もできない。実質的には**作成の瞬間だけ使われて以降は不要**になる。

**違い②: SolanaにはもっとCAらしい「PDA」という仕組みもある**

Solanaには**PDA (Program Derived Address)**という、秘密鍵が存在しないアドレスを作る仕組みもある。これはEthereumのCAに**もっと忠実に対応する**:

- プログラムID + seed文字列 から数学的にアドレスを導出
- 秘密鍵が最初から存在しない
- そのアドレスで"署名"できるのはプログラム本体のみ（seedを知っている前提で）

ここでいう **プログラムID** は、そのオンチェーンプログラム自身の公開鍵アドレス（Ethereumでいうコントラクトアドレスに近い識別子）。  
**seed文字列** は、PDAを一意に決めるために開発者が渡す材料（例: `"profile"`, ユーザーの公開鍵, `mint.address` など）で、同じプログラムID + 同じseedの組み合わせなら常に同じPDAが導出される。

NFTのメタデータ、Anchor製プログラムの状態アカウント、ATA（Associated Token Account）などは典型的にPDAを使う。今日の Mint が PDA ではなく普通のキーペア方式だったのは、**動画の入門段階で分かりやすさを優先した**のと、Mintは別にPDAでなくても困らないから。

##### 3種類のアドレスの対応表

これでSolanaのアドレス理解が整理できる:

| Solana | Ethereum 対応 | 秘密鍵 | 例 |
|---|---|---|---|
| **ユーザーウォレット**（Keypair） | **EOA** | あり（人間が管理） | Phantom Wallet、今日の `feePayer` |
| **通常アカウント**（Keypair由来）| **CA に近い**（実装は違う） | あり（ただし作成時のみ使用） | 今日の `mint.address` |
| **PDA**（Program Derived Address） | **CA と最も近い** | なし（プログラムだけが扱える） | ATA、Anchor state account |

##### まとめ

- **`feePayer` = 開発者のEOA相当**（今日の例では使い捨てキーペアだが、実務ではユーザーのウォレット）
- **`mint.address` = EthereumのCA相当**のデータアカウント。実装上はキーペアから生まれるが、作成後の秘密鍵は事実上不要で、書き換えは Token-2022 プログラム経由
- **より厳密にCAに対応するのはPDA**。こちらは次回以降のATA作成の回で触れることになるはず

この対応関係を押さえると、「今日のスクリプトで generateKeyPairSigner を2回呼んでいる」のが **「EOA1つ + CA相当1つ」を作っていた**、と自然に読めるようになる。

#### Q: 作成後に事実上使わない秘密鍵を、そもそも作る意味はあるのか？

「作成後に不要になる秘密鍵なら、最初から作らなくていいのでは？」というのは当然の疑問である。実際、**秘密鍵を持たないアドレス（= PDA）だけで全てを設計する**ことも技術的には可能。しかしキーペア方式を残している（そして多くの場面で使われている）のには、ちゃんとした理由が複数ある。

##### 理由①: 作成時の「本人同意の署名」が必要

Solanaランタイムには**「新しく作られるアカウントは、そのトランザクションの署名者でなければならない」**というルールがある。これは以下のような攻撃を防ぐためのセキュリティ機構:

- 悪意ある誰かが、予測可能な未使用アドレス（例: `hash(userPubkey)` で決めたプロフィール用アカウント）を先回り作成し、想定外データを書き込んで初期化を失敗させる
- 本来ユーザー本人が初回作成するはずのアカウントを第三者が先に埋めてしまい、`already in use` などで正規フローを妨害する

新しいアカウントが「自分が作られることに同意した」署名を付けることで、**そのアドレスの所有者（= 秘密鍵を持つ人）だけがそこに新アカウントを作れる**ことを保証している。

この「作成時の署名」のために、**秘密鍵が1回だけ必要**になる。作成後は使わなくなるが、作成の瞬間には重要な役割を果たしている。

> **PDAの場合**: PDAは秘密鍵を持たないが、代わりに**所有プログラムが `invoke_signed` で seed を使って署名**する、という特別な仕組みで同じルールを満たす。つまり「誰かが署名する」というルール自体は同じで、署名の仕方が違うだけ。

##### 理由②: アドレスの衝突回避とユニーク性

Ed25519 の鍵空間は 2^256 に近い広さで、**ランダム生成したアドレスが他と衝突する確率は天文学的に小さい**。つまり:

- 何も考えずに `generateKeyPairSigner()` を呼べば、世界中の誰とも被らないアドレスが手に入る
- **seed を何にするか悩む必要がない**

PDAだと「何を seed にするか」を設計時に決める必要がある。Mintのような"単に一意のトークン識別子が欲しいだけ"のケースでは、むしろランダムキーペアの方が素直。

##### 理由③: バニティアドレス（好きな文字で始まるアドレス）

キーペア方式だと、**総当たりで鍵を生成し続けて、好みのアドレスが出るまで回す**ことができる。例えば:

- USDCのMintアドレスは `EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v` で、末尾 `Dt1v` などは意味を持たないランダム
- 一方、プロジェクトによっては「`COIN...`で始まるアドレス」のようなバニティ（見栄え用）アドレスを使うことがあり、これは**ランダム生成を何億回も繰り返して一致する鍵を探す**ことで実現する（`solana-keygen grind` コマンドが有名）

これは PDA では不可能（PDAは seed から決定的に決まってしまうため、自由に選べない）。**キーペア方式ならではの利点**。

##### 理由④: 歴史的経緯と使い分けの自由度

SolanaはローンチときからKeypair方式が先にあり、**PDAは後から追加された仕組み**。既存コードやドキュメント、教育資料の多くが Keypair 方式で書かれており、互換性のため残っている。

また、使い分けの設計指針としては:

| アドレスの使い方 | 適しているもの |
|---|---|
| 「このMintは独立した一意の識別子として存在する」 | **Keypair**（今日のMint） |
| 「このアカウントはXさんのYに対応する、決定的に導出される」 | **PDA**（次回触るATA） |
| 「プログラム自身が後から署名する必要がある」 | **PDA**（プログラムの状態アカウント） |

**「一意性だけあれば十分」なものにはKeypairが自然で、「ユーザーやプログラムから一意に導けること」が重要なものにはPDAが自然**、という棲み分け。

##### 理由⑤: 事前に第三者が生成したアドレスを渡せる

キーペアはあらかじめ**別のマシン・別の人間が生成しておくことができる**。例えば:

- セキュアなオフライン環境で生成 → そのアドレスだけを他の人に伝える
- プロジェクトチームの別メンバーが "このアドレスに作って" と指定する
- ハードウェアウォレット上で生成

PDAだとアドレスはプログラムIDと seed から自動計算されるので、このような「**誰かが先に鍵を作っておいて後で使う**」というワークフローが取りにくい。

##### 結論

作成後に不要になる秘密鍵を作るのは**無駄ではない**。

- **作成の瞬間の本人同意署名**に必須
- **グローバル一意なアドレス**を簡単に得られる
- **バニティアドレス**を作れる
- **第三者が事前に生成**できる自由度

これらの価値のために「一瞬だけ使われる秘密鍵」が存在している、と理解すると腑に落ちる。逆に言えば、**これらのメリットが不要で、導出ルールが決まっているほうが嬉しい**場面（= ATA、Anchor state account、NFT metadata等）では、PDAを選ぶのが自然。

Solanaの設計は「**2つの選択肢を用意して、開発者が場面に応じて選ぶ**」という思想であり、キーペア方式とPDAは**どちらかが優れているわけではなく、用途が違う**というのが正しい理解になる。

### 4.6 命令①: システムプログラムで空アカウントを作成

```typescript
const createAccountInstruction = getCreateAccountInstruction({
  payer: feePayer,
  newAccount: mint,
  lamports: rent,
  space,
  programAddress: TOKEN_2022_PROGRAM_ADDRESS,
});
```

Solanaでは「Mintアカウント作成」は**2段階**で行う必要がある:

1. **システムプログラム**で「Token-2022プログラム所有の空アカウント」を作る ← これがここ
2. **Token-2022プログラム**で「そのアカウントをMintとして初期化」する ← 次の4.7

`getCreateAccountInstruction` は、システムプログラムの `CreateAccount` 命令を呼び出すための**トランザクション命令オブジェクト**を組み立てる。

引数の意味:

- **`payer`**: アカウント作成のコスト（= rent）を払うアカウント。`feePayer` が負担する。
- **`newAccount`**: 作る先の新しいアカウント（= `mint`）。こちらも署名者として必要（本人が作られることに同意する署名）。
- **`lamports: rent`**: 新アカウントに初期入金する額。ここでrent-exempt値を入れるので永久に死なない。
- **`space`**: 新アカウントが使うデータ領域のバイト数（Mintサイズ）。
- **`programAddress: TOKEN_2022_PROGRAM_ADDRESS`**: 新アカウントの**オーナー（所有プログラム）**。デフォルトだとシステムプログラム所有になるが、Mintは**Token-2022プログラムが所有している必要がある**ので明示的に指定する。

> **所有者（owner）の概念**: Solanaの全アカウントはどれか1つのプログラムに所有されており、そのアカウントのデータを書き換えられるのは所有プログラムだけ。Mintは当然Token-2022が書き換える必要があるので、最初からオーナーをTOKEN_2022に設定して作る。

### 4.7 命令②: Token-2022プログラムでMintを初期化

```typescript
const initializeMintInstruction = getInitializeMintInstruction({
  mint: mint.address,
  decimals: 9,
  mintAuthority: feePayer.address,
});
```

前のステップで作った「空のアカウント」を、Token-2022プログラムが読んで**Mintとしての中身を書き込む**命令。

引数の意味:

- **`mint: mint.address`**: 初期化対象のMintアカウントのアドレス。
- **`decimals: 9`**: **トークンの小数点以下の桁数**。SOLと同じ9桁にしている。つまり最小単位 = 10^-9 トークン。USDCは6桁、NFTは0桁など、用途に応じて変える。
- **`mintAuthority: feePayer.address`**: **このトークンを発行（mint）できる権限**を誰が持つか。`feePayer` にしているので、今回生成したこの鍵の持ち主だけが後から「追加発行」できる。

> **decimals の意味（素人向け）**: `decimals: 9` と指定すると、トークン量は内部的には整数で管理されるが、表示上は「整数値 ÷ 10^9」と解釈される。例えば内部値が `1_500_000_000` なら、ユーザーには `1.5 トークン` と見える。USDCのように「少額決済が多い」通貨は6桁、ビットコイン風なら8桁、など用途で選ぶ。

### 4.8 命令を配列にまとめる

```typescript
const instructions = [createAccountInstruction, initializeMintInstruction];
```

上で作った**2つの命令**を1つの配列にする。Solanaでは**1つのトランザクションに複数の命令を含められる**ため、この2つを**アトミック（どちらも成功するか、どちらも失敗するか）**に実行できる。

### 4.9 最新のブロックハッシュを取得

```typescript
const { value: latestBlockhash } = await rpc.getLatestBlockhash().send();
```

Solanaのトランザクションには**有効期限**があり、「このブロックハッシュの時点で有効なTXである」と指定する必要がある。ブロックハッシュは約1〜2分で失効するため、送信直前に最新の値を取る。

### 4.10 トランザクションメッセージの組み立て（pipe）

```typescript
const transactionMessage = pipe(
  createTransactionMessage({ version: 0 }),
  (tx) => setTransactionMessageFeePayerSigner(feePayer, tx),
  (tx) => setTransactionMessageLifetimeUsingBlockhash(latestBlockhash, tx),
  (tx) => appendTransactionMessageInstructions(instructions, tx),
);
```

> **読み方の補足（引用コードは `tx` のまま）**: 実務では各ステップで同じ名前 `tx` にそろえることが多い。流れを追いやすくするだけなら、**意味は変わらず**次のように **ステップごとに別名**にしてもよい（いずれも「`pipe` が前段の戻り値を、次のアローの第1引数に渡す」だけである）。
>
> - 1行目のアロー: `(tx1) => setTransactionMessageFeePayerSigner(feePayer, tx1)` と書くと、`tx1` には **`createTransactionMessage` の戻り値**（空メッセージ）が入る。
> - 2行目: `(tx2) => setTransactionMessageLifetimeUsingBlockhash(latestBlockhash, tx2)` と書くと、`tx2` には **直前の `setTransactionMessageFeePayerSigner` の戻り値**が入る。
> - 3行目: `(tx3) => appendTransactionMessageInstructions(instructions, tx3)` と書くと、`tx3` には **その直前の `setTransactionMessageLifetimeUsingBlockhash` の戻り値**が入る。

**`pipe` と `(tx) =>` の読み方**（ここが一番つまずきやすい）:

`@solana/kit` の `pipe(x, f, g, h)` は **`h(g(f(x)))` と同じ**で、「最初の値 `x` に、順に関数をかけていく」イメージ。各 `f, g, h` は **1引数の関数**として呼ばれる。

```typescript
function pipe(initial, fn1, fn2, fn3) {
  const after1 = fn1(initial);
  const after2 = fn2(after1);
  return fn3(after2);
}
```

- **`createTransactionMessage({ version: 0 })` の戻り値**が上の `initial` に相当する。これは **`tx` という変数への代入ではない**（`pipe` が実行されて初めて、次のアローの第1引数に渡される）。
- **`(tx) => setTransactionMessageFeePayerSigner(feePayer, tx)`** は、`fn1(initial)` と呼ばれたときに **`tx` に `initial` が束縛される**アロー関数（`message` など名前は任意）。ソースを上から読んだ瞬間に `tx` が埋まるのではなく、**`pipe` 実行時の `fn1(initial)` という呼び出し**で初めて値が入る。通常の関数式なら次と同義:

```typescript
function (tx) {
  return setTransactionMessageFeePayerSigner(feePayer, tx);
}
```

- **`pipe` が渡すのは常に1値**なので、次のステップに渡せるのも **1個の値（ここではトランザクションメッセージ）だけ**。ところが `setTransactionMessageFeePayerSigner` は引数が2つ（`feePayer` とメッセージ）必要なので、そのままでは `pipe` に渡せない。
- そこで **`(tx) => setTransactionMessageFeePayerSigner(feePayer, tx)` のようにアロー関数で包む**。`pipe` がこの関数を呼ぶときは **`fn1(initial)` の形**なので、**左側の `tx` に入るのが `initial`（= `createTransactionMessage` の戻り値）** である。外側のスコープにすでにある `feePayer` をそのまま式に書き込み（**クロージャ**で「外から閉じ込める」＝アロー関数の中から参照する）、**`pipe` から渡ってきた1個だけを第2引数の `tx` に載せる**。第1引数の `feePayer` は `pipe` が渡したわけではなく、**あなたがコードの上で決めた固定の値**。
- **「流れてくる値は必ず最後の引数」といった共通ルールはない**＝`pipe` が「常に第2引数に流す」と決めているわけではない、という意味。別のAPIなら `(x) => foo(x, other)` のように **第1引数に流す**こともできる。今回は `setTransactionMessageFeePayerSigner` の定義が `(feePayer, message)` 型なので **第2引数に載せているだけ**。

1. **`createTransactionMessage({ version: 0 })`**: 空のトランザクションメッセージを作成。`version: 0` は **バージョン付きメッセージの v0 形式**を使うという指定（レガシー形式ではない）。

   **レガシーとバージョン付きの違い**は、同じ「トランザクション」でも **メッセージの封筒の形（エンコードのルール）が違う**イメージである。

   | | レガシー | バージョン付き（v0） |
   |---|---|---|
   | 歴史 | 最初からあった古い標準 | 後から追加された形式 |
   | バージョン | 先頭に番号なし（暗黙に1種類） | 先頭に `0` など |
   | アカウントの載せ方 | 本文にフルで並べる前提 | v0 は **Address Lookup Table（ALT）** を参照して短く書ける（多アカウント向け） |

   v0 は **ALT を必須にするわけではない**。ALT をまだ使わなくても v0 で組み立てるのが `@solana/kit` では普通で、今回のスクリプトもそのパターン。`version: 0` = 「将来・必要なら ALT も載せられる、新しい封筒でメッセージを作る」。
2. **`setTransactionMessageFeePayerSigner(feePayer, tx)`**: 手数料の支払者を `feePayer` に設定。
3. **`setTransactionMessageLifetimeUsingBlockhash(latestBlockhash, tx)`**: 有効期限（ライフタイム）を先ほど取得したブロックハッシュでセット。
4. **`appendTransactionMessageInstructions(instructions, tx)`**: 命令2つを追加。

各ステップは**前の結果を受け取って新しいトランザクションメッセージを返す**（破壊的変更ではなくイミュータブル）ので、最終結果がそのまま `transactionMessage` に入る。

### 4.11 署名

```typescript
const signedTransactionMessage =
  await signTransactionMessageWithSigners(transactionMessage);
```

トランザクションメッセージに含まれる**全ての署名者**（= `feePayer` と `mint`）の秘密鍵で自動的に署名する。`generateKeyPairSigner` で作ったキーペアは署名機能を内蔵しているので、ここで署名情報が取り出せる。

### 4.12 型アサーション（`lifetimeConstraint` の補完）

```typescript
const signedTransactionWithLifetime =
  signedTransactionMessage as typeof signedTransactionMessage & {
    lifetimeConstraint: {
      lastValidBlockHeight: bigint;
    };
  };
```

これは `@solana/kit` の**型定義のクセ**を吸収するための追加行である。`signTransactionMessageWithSigners` の戻り値型に `lifetimeConstraint.lastValidBlockHeight` が含まれないケースがあり、続く `sendAndConfirmTransactionFactory` がそのプロパティを要求するため、**「あるよ」と宣言**してあげている。

`X as typeof Y` の読み方は「**XをYと同じ型として扱う**」。ここでは `signedTransactionMessage as typeof signedTransactionMessage` なので、「この値を元の `signedTransactionMessage` と同じ型として扱う」という意味になる。さらに `& { lifetimeConstraint: ... }` を足しているので、最終的には「元の型 + `lifetimeConstraint.lastValidBlockHeight` がある型」として扱わせている。なお `as` は型チェック上の宣言であり、実行時にオブジェクトへプロパティを追加する処理ではない。

実行時には確かに値が入っているので、TypeScriptのコンパイルを通すための"おまじない"と理解してよい。将来のライブラリ更新で不要になる可能性が高い。

### 4.13 送信と確認

```typescript
await sendAndConfirmTransactionFactory({ rpc, rpcSubscriptions })(
  signedTransactionWithLifetime,
  { commitment: "confirmed" },
);
```

`airdropFactory` と同じ2段階呼び出しのパターン。**rpc/wsを渡して関数を作り、即座にその関数にトランザクションを渡す**。

この関数は裏で以下を行っている:

1. 署名済みトランザクションをRPCに送信
2. WebSocketで「このトランザクションの状態」を待ち受け
3. `commitment: "confirmed"` に達したら成功としてPromiseを解決

### 4.14 トランザクション署名の取得と表示

```typescript
const transactionSignature = getSignatureFromTransaction(
  signedTransactionWithLifetime,
);

console.log("Mint Address:", mint.address);
console.log("Transaction Signature:", transactionSignature);
```

- **`getSignatureFromTransaction`**: 署名済みトランザクションから**TX署名（ブロックチェーンエクスプローラで検索可能な一意ID）**を取り出す。
- 最後にMintアドレスと署名を出力して終了。

### 4.15 メイン関数のエラーハンドリング

```typescript
main().catch((err) => {
  console.error(err);
  process.exit(1);
});
```

`main()` が例外を投げた場合にコンソールにエラーを出して非ゼロ終了する、定番の形。流れを分解すると:

- **`main()`** は `async` 関数なので戻り値は `Promise`。正常終了なら `resolve`、失敗なら `reject` される。
- **`.catch((err) => { ... })`** は、その `reject` を最終的に受け取る受け皿。これがないと「Unhandled Promise Rejection（未処理のPromiseエラー）」になり、原因が追いにくくなる。
- **`console.error(err)`** でスタックトレース付きのエラー内容を標準エラー出力へ出す。どこで失敗したかを確認しやすい。
- **`process.exit(1)`** は「異常終了」を明示する終了コード。シェルスクリプトやCIでは `0` が成功・`1` 以上が失敗なので、自動実行時にも失敗を正しく検知できる。

## 6. 全体の流れを図で整理

```
[feePayer 鍵ペア生成] → [1 SOL airdrop]
                              ↓
[mint 鍵ペア生成] → [getMintSize でサイズ取得]
                              ↓
[getMinimumBalanceForRentExemption で必要SOL取得]
                              ↓
[命令①: CreateAccount (System)]
  ├─ payer: feePayer
  ├─ newAccount: mint
  ├─ lamports: rent
  ├─ space: mintSize
  └─ owner: TOKEN_2022_PROGRAM_ADDRESS
                              ↓
[命令②: InitializeMint (Token-2022)]
  ├─ mint: mint.address
  ├─ decimals: 9
  └─ mintAuthority: feePayer.address
                              ↓
[最新 blockhash 取得]
                              ↓
[pipe で TransactionMessage 組み立て]
  createTransactionMessage({version:0})
    → setFeePayer(feePayer)
    → setLifetime(blockhash)
    → appendInstructions([命令①, 命令②])
                              ↓
[signTransactionMessageWithSigners で署名]
                              ↓
[sendAndConfirmTransaction で送信・確認]
                              ↓
[Mint Address と Signature を出力]
```

## 7. 今回学んだキーコンセプト

1. **Mintアカウントは「空アカウントを作る」→「Mintとして初期化する」の2ステップ**
   - `System::CreateAccount` と `Token-2022::InitializeMint` を**同一トランザクション**に入れることで、アトミックに処理する。片方だけ成功して片方が失敗、という中途半端な状態にならない。

2. **`@solana/kit` は関数型パイプラインスタイル**
   - `pipe()` で「空のトランザクション → 支払者を設定 → 有効期限を設定 → 命令を追加」と積み上げる。オブジェクト指向の "tx.setFeePayer()" スタイルではない。
   - Factory パターン（`airdropFactory({rpc, ws})(...)`）も多用される。設定を固定した関数を作ってから呼ぶ。

3. **Rent-exempt は Solana の必須作法**
   - どのアカウントも、サイズ相当の最低SOL残高を入れておかないと勝手に消滅する。`getMinimumBalanceForRentExemption` で計算して確実に入れる。

4. **decimals はトークンの「桁数」を決める、後から変えられない設計パラメータ**
   - 9桁（SOLと同じ）、6桁（USDC）、0桁（NFT）など、用途で決める。
   - 内部的には整数、表示上は `10^decimals` で割る、という仕組み。

5. **Token-2022 は新しい標準**
   - 旧SPL Tokenの後継。`@solana-program/token-2022` / `TOKEN_2022_PROGRAM_ADDRESS` を使うのが今時の書き方。

## 9. RustとTypeScriptの役割分担（なぜ今日はRustが要らなかったのか）

「Solanaのプログラムは Rust で書く」というイメージは**半分正しく、半分ミスリーディング**である。ここで一度整理しておく。

### 9.1 Solanaの世界は「オンチェーン」と「オフチェーン」の2層に分かれる

Solanaのシステムは、大きく以下の2つの層で成り立っている:

| 層 | 実行場所 | 言語 | 役割 |
|---|---|---|---|
| **オンチェーン**（プログラム / スマートコントラクト） | Solanaバリデーター上 | **Rust**（または C, Pinocchio等） | チェーン上のデータ（アカウント）を書き換えるロジック |
| **オフチェーン**（クライアント） | 自分のPC、サーバー、ブラウザ | **TypeScript** / Python / Rust / Goなど何でも | オンチェーンのプログラムを**呼び出す**コード、UI、バックエンド |

つまり「Solanaは Rust」というのは正確には**「オンチェーンで走るプログラムは Rust（が主流）」**という意味で、**オフチェーンのクライアント側は TypeScript で全く問題ない**。むしろWeb開発と統合するので TypeScript が主流。

### 9.2 オンチェーンプログラム（Rust側）が担うもの

Rustで書かれたオンチェーンプログラムは、**「チェーン上のアカウントのデータをどう書き換えるか」のルール**を定義する。例えば:

- 前回（4/11）の **Counter プログラム**: 「`Update` 命令が来たら counter.count を +1 する」というロジック
- 3/15の **Crowdfunding プログラム**: 「寄付があったら total_donated に加算する」というロジック
- **System プログラム**（Solana組み込み）: 「CreateAccount 命令が来たら新しいアカウントを作って lamports を割り当てる」
- **Token-2022 プログラム**（Solana組み込み）: 「InitializeMint 命令が来たら Mint の decimals と authority を書き込む」

**重要なポイント**: Rustプログラムは**チェーン上にデプロイされて初めて使える**。デプロイされた後はただの「呼べるブラックボックス」であり、オフチェーンからは「どんな命令が使えるか」「どの引数を渡すか」だけ知っていれば呼べる。

### 9.3 オフチェーンクライアント（TypeScript側）が担うもの

TypeScriptが担うのは以下のようなこと:

- どのプログラムを呼ぶか決める（アドレス指定）
- どの命令を使うか選ぶ（例: `CreateAccount`, `InitializeMint`）
- 命令の引数を組み立てる（payer, space, decimals など）
- 複数の命令をトランザクションにまとめる
- 署名する
- RPCに送信する
- 結果を受け取って表示する

つまりTypeScriptは**「プログラムへのリモコン」**であり、**ロジックそのものは持たない**。今日のスクリプト内の `getCreateAccountInstruction(...)` は**ただの"命令データの組み立て"**であって、実際に"空アカウントを作る"処理をしているのはチェーン上の System プログラム（Rust製）である。

### 9.4 「プログラムのコードが1行も出てこない」のはなぜか

今日のスクリプト `new-token.ts` には `impl`、`pub fn`、`Context`、`declare_id!` のような Rust らしい記述が一切ない。これは**新しいオンチェーンプログラムを書いていないから**。

今日使った System プログラムと Token-2022 プログラムは、**Solana のコア開発者が Rust で書いて、すでにチェーン上にデプロイ済み**の「ビルトインプログラム」である。アドレスも固定で公開されていて、誰でも `TOKEN_2022_PROGRAM_ADDRESS` のような定数で参照できる。

比喩で言うと:

- **Rustでプログラムを書く** = 新しい「冷蔵庫」を作って家電量販店（= チェーン）に納品する
- **TypeScriptでクライアントを書く** = すでに家電量販店に並んでいる冷蔵庫を買ってきて、中身を出し入れする

今日は**冷蔵庫を自作せず、Solana公式が出荷済みの冷蔵庫（Token-2022）を使っただけ**なので、Rustは1行も要らなかった。

### 9.5 前回（4/11）との対比

| 項目 | 4/11 (Counter DApp) | 4/20 (Token Mint) |
|---|---|---|
| 新しいオンチェーンロジックが必要か？ | **必要**（counter.count をカウントアップする独自ロジック） | **不要**（Mintの初期化はToken-2022が提供済み） |
| Rustコード | `lib.rs` に `increment` / `decrement` などを実装 | **一切書かない** |
| `anchor build` / `anchor deploy` | 必要（自作プログラムをデプロイ） | **不要** |
| TypeScript側の役割 | 自作プログラムを呼ぶ | **既存のビルトインプログラムを呼ぶ** |
| 使うプログラムID | 自分でデプロイ後に発行されたID | Solanaが決めた固定ID（`TOKEN_2022_PROGRAM_ADDRESS`） |

### 9.6 いつRustが必要で、いつTypeScriptだけでいいのか

ざっくりしたガイドライン:

- **Rust（オンチェーンプログラム）を書く必要があるケース**
  - チェーン上で独自のビジネスロジックを実行したいとき
  - 例: 独自のカウンター、クラウドファンディング、ゲームの勝敗判定、DEXのスワップロジック、NFTミント時の独自ルール等
- **TypeScriptだけで十分なケース**
  - 既存のビルトインプログラム（System / Token-2022 / Stake / Vote等）だけで完結する操作
  - 既存の3rdパーティプログラム（Metaplex NFT, Jupiter, Raydium等）を呼ぶだけのケース
  - 例: 今日のような**素のトークン発行**、トークン送金、NFTメタデータ作成（Metaplexを使うなら）等

**今日の「新しいトークンを作る」は完全に後者のケース**で、Solana公式がRustで書いたToken-2022をTypeScriptから呼ぶだけで完結する。これが「Rustが要らない」ことの正体である。

> **補足**: 次回以降予定している Token Account 作成、ATA作成、Mint Token、Transfer、Freeze、Thaw、Burn、Close も**全部 Token-2022 ですでに提供されている命令**なので、**Rustは最後まで要らない**。素のSPL系操作だけなら、実務でもTypeScriptだけで組み上がる。独自ロジックを挟みたくなったタイミングで初めてRust（Anchor / Pinocchio）を書くことになる。

### 9.7 Ethereum（ERC20）との対比 — なぜSolanaはスマートコントラクトを書かずにトークンが作れるのか

「でもEthereumでは、新しいトークンを作るたびにSolidityで ERC20 のコントラクトを書いてデプロイするよね？」という疑問が自然に出る。ここが **Ethereum と Solana で根本的に設計思想が違う最大のポイント**なので、腰を据えて比べる。

#### Ethereumの世界（ERC20）

Ethereumで新しいトークンを作るときの流れは:

1. Solidityで **ERC20 の**`transfer`、`balanceOf`、`approve` などの関数を**自分で実装**したコントラクトを書く
2. `forge create` / Hardhatなどでそのコントラクトを**ブロックチェーンにデプロイ**
3. 各ユーザーの残高は、そのコントラクト内部の `mapping(address => uint256) balances` に記録される

つまり**トークンごとに1つのスマートコントラクトが新しく増える**。USDCコントラクト、DAIコントラクト、UNIコントラクト、…と別々の実装が何百万個もチェーン上に並んでいる。

- `transfer` のロジックは各コントラクトが独自に持つ（バグがあれば個別に脆弱）
- 残高データもコントラクト内部に閉じて保持
- ERC20 は**「インターフェース規約」**にすぎず、実装そのものではない

#### Solanaの世界（Token-2022）

Solanaは発想が全く逆である:

1. **Token-2022 プログラム**が1つだけ、Solana公式によってRustで書かれチェーンにデプロイされている
2. 新しいトークンを作るとは、そのToken-2022プログラムに**「新しい Mint Account を作って」と命令を送る**だけ
3. 各ユーザーの残高は、**Token Account**（別の独立したアカウント）に記録され、Token-2022 がそれを読み書きする

つまりSolanaでは、トークンが増えても**プログラムは1つのまま**で、増えるのは**データ（アカウント）**だけ。USDCも、ゲーム内通貨も、今日作ったトークンも、**全員が同じ Token-2022 プログラムを共有**している。

- `transfer` のロジックは Token-2022 が一箇所で持つ（バグがあれば全トークン一斉修正、逆に言えば一箇所が正しければ全トークン安全）
- 残高はアカウント単位で独立して存在し、プログラムはステートレスに近い
- 規約と実装が一体化している（= ERC20のような「規約だけある」状態が存在しない）

#### 対比表

| 観点 | Ethereum (ERC20) | Solana (Token-2022) |
|---|---|---|
| トークンを新規発行するときに書くもの | **Solidityコントラクトを新規作成** | **書かない**（既存プログラムに命令を送るだけ） |
| トークンごとのコントラクト数 | 1トークンにつき1コントラクト | 全トークンで1プログラムを共有 |
| 残高の保存場所 | コントラクト内の `mapping` | 独立した Token Account |
| `transfer` のロジック | 各コントラクトが独自実装 | Token-2022 が唯一実装 |
| バグ発見時 | 個別コントラクトごとに修正必要 | Token-2022 を直せば全トークンに反映 |
| "規約"と"実装"の関係 | ERC20は**規約のみ**（各自実装） | Token-2022は**規約＝実装**（1つしか存在しない） |
| 開発に必要な言語 | **Solidity** 必須 | **TypeScript のみ**で完結（独自ロジックなし時） |
| デプロイコスト | 1コントラクトごとに発生（ガス代大） | 発生しない（命令送信のみ） |

#### なぜSolanaはこの設計にできたのか

この違いは「どちらが優れているか」というより、**アーキテクチャの根本思想の違い**から来ている:

- Ethereumは **「コードとデータが一体のコントラクト」** モデル。コントラクトは自分の状態を内部に持つ（オブジェクト指向的）。
- Solanaは **「コードとデータが分離したプログラム＋アカウント」** モデル。プログラムはロジックだけ、データは外部のアカウントに置く（関数型的）。

後者のモデルだと、**「同じロジックで動く別々のデータ」** を自然に表現できる。Token-2022 というロジック1つと、Mint Account / Token Account というデータ多数、という構造で全トークンが表現できる。

> **実感**: 今日のスクリプトで `getInitializeMintInstruction({mint, decimals, mintAuthority})` と書いた時点で、ERC20で言えば `new MyToken("MyToken", "MTK", 18)` を**コントラクトをデプロイせずに**やっているのと同義になる。Ethereumだと絶対にSolidityを書かないとできないことが、Solanaでは**既存プログラムへの命令1つで済む**。これがSolanaで「同じトークン作成でも開発コストが劇的に低い」と言われる理由である。

逆に言えば、**Ethereumで言う「ERC20コントラクトに独自のロジックを埋め込む」ような自由度**（例: `transfer` 時に手数料を取る、ブラックリストを持つ、等）は、Solanaでは:

- Token-2022 の**拡張機能（Extension）**として用意されているものを使う（Transfer Fee, Default State, 等）
- それでも足りなければ、**別プログラムをRustで書いて**間に挟む（CPI: Cross-Program Invocation）

という形で実現する。つまり**「独自ロジックが必要な瞬間に初めてRustが必要になる」**という先ほどの 9.6 の話に戻ってくる。

### 9.8 Token-2022 は「既にデプロイ済み」— 確認すべきよくある疑問

ここまでの説明で「Token-2022 プログラムは既にオンチェーンにデプロイされている、という理解で合っているのか？」と気になるのは自然な反応である。答えは **はい、その通り** で、これは Solana 開発を正しく理解する上で非常に重要な前提なので、念押しで整理しておく。

#### 事実関係

- **プログラムIDは全クラスター共通で固定**: `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb`
  - 今日のスクリプトで使った `TOKEN_2022_PROGRAM_ADDRESS` はこの定数を単にエイリアスしたもの
  - mainnet、devnet、testnet、Localnet のどこで動かしても**同じアドレス**
- **Localnet（Surfpool / solana-test-validator）にも最初から組み込まれている**
  - だから今日、自分で Token-2022 をデプロイする作業をしなくても `getInitializeMintInstruction` で命令を投げれば即動いた
  - もしも未デプロイだったら「プログラムが見つからない」エラーになるはず
- **USDC も、Jupiter で扱われるあらゆるトークンも、今日作った自分のトークンも、全員この1つの Token-2022 を共有している**
  - チェーン上に Token-2022 のインスタンスが複数あるわけではない、**唯一の実体**

#### Ethereumとの対比で言うと

Ethereumで ERC20 トークンを作る場合「ERC20コントラクトをデプロイする」作業が**各開発者の責任**になるが、Solanaではそれに相当する作業を**Solanaコアチームが全クラスターで事前に済ませてくれている**。開発者がやるのは「そのプログラムに命令を送って Mint Account を1個新設する」ことだけ。

#### 他にも事前デプロイ済みの「ビルトインプログラム」がある

Token-2022 だけでなく、以下のような基礎プログラムも全クラスターで事前デプロイ済み:

| プログラム | プログラムID | 役割 |
|---|---|---|
| System Program | `11111111111111111111111111111111` | アカウント作成、SOL転送 |
| SPL Token（旧版） | `TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` | 旧トークン規格（互換目的で残っている） |
| Token-2022 | `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` | 新トークン規格（今日使ったのはこれ） |
| Associated Token Account Program | `ATokenGPvbdGVxr1b2hvZbsiqW5xWH25efTNsLJA8knL` | ATAの導出・作成 |
| Stake Program | `Stake11111111111111111111111111111111111111` | SOLのステーキング |
| Vote Program | `Vote111111111111111111111111111111111111111` | バリデーター投票 |

今日のスクリプトで `getCreateAccountInstruction` から呼び出していた **System Program** も、この表の1行目である。これらは Solana のランタイムに組み込まれている、いわば **「OSの標準ライブラリ」** のような存在で、TypeScript から命令を送るだけで利用できる。

次回以降触る **ATA** も Associated Token Account Program が事前デプロイ済みなので、同じくクライアントから命令を投げるだけで作成できる。**最後までRustが要らない**と断言できるのは、これらのインフラが全部すでにチェーン上に揃っているからである。

## 10. まとめ

今回の学習で、Solana上に**自分だけの新しいトークン（Mintアカウント）をローカルチェーンに発行**する、ゼロから手書きのTypeScriptスクリプトを完成させた。

```
[npm init] → [依存インストール] → [type:module に変更] →
[Surfpool 起動] → [new-token.ts を書く] → [npx tsx で実行] →
[Mint Address を取得]
```

次回以降は、今日作った Mint に対して、**トークンのライフサイクル全体**を一通り触っていく予定:

- **Token Account の作成** — 特定のMintのトークンを保有するための専用アカウントを作る
- **Associated Token Account（ATA）の作成** — ウォレットアドレスから一意に導出される"標準的な"Token Accountを作る
- **Mint Token** — Mint Authorityの権限で、指定量のトークンをToken Accountに鋳造する
- **Transfer Token** — あるToken Accountから別のToken Accountへトークンを送る
- **Freeze Token** — Freeze Authorityの権限で、Token Accountを凍結し送金・操作を禁止する
- **Thaw Token** — 凍結したToken Accountを解除し再び使えるようにする
- **Burn Token** — Token Accountのトークンを焼却（永久破棄）し、流通量を減らす
- **Close Token Account** — 残高ゼロのToken Accountを閉じ、rent分のSOLを回収する

**今日のスクリプトで作った Mint Address が、これらすべての操作の起点**になる。Mint（= トークンの発行元）は1つだが、それを保有するToken Accountは無数に存在しうる — その関係性を手を動かして理解していく。

日本語Bootcampは想像以上に分かりやすく、最新の `@solana/kit` 世代のAPIに沿って書かれているので、ドキュメントの古さで詰まる心配もない。このペースで続けていく。
