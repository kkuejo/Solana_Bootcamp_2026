# Token Account と Associated Token Account を作る

**日付**: 2026年4月28日
**学習内容**: 前回（4/20）作った Mint に対して、**Token Account** と **Associated Token Account（ATA）** をそれぞれ作るスクリプトを書いた。Token-2022 プログラムが提供する `InitializeAccount2` 命令と、Associated Token Account Program の `CreateAssociatedToken` 命令を使い、両者の違いを実装の差から理解する。引き続き `@solana/kit` + `@solana-program/token-2022` を使用。

## 0. 本日の位置づけ

前回（4/20）は **Mint Account**（= トークンの発行元アカウント）を作成した。Mint があるだけでは「このトークン、こういう仕様で存在しますよ」と宣言しただけで、**誰かが実際にそのトークンを持つ**ためには別途**保管庫となるアカウント**が必要になる。

その保管庫が **Token Account** であり、その「標準的な作り方」として用意されているのが **Associated Token Account（ATA）** である。本日は同じスクリプトの中で**両方の作り方**を実装し、何が違うのかを実装レベルで把握した。

> **前回からの連続性**: 4/20で `mint = await generateKeyPairSigner()` し、`InitializeMint` で初期化した Mint アドレスがそのまま今日の主役になる。Mint は「トークンの設計図」、Token Account は「その設計図に基づくトークンを保管する財布」という関係。

## 1. Token Account と Associated Token Account の違い

まず**今日の核心となる概念整理**を先にやる。コードを読み始める前に頭に入れておくと、後の実装の差が一発で分かる。

### 1.1 ざっくり一言で

| 項目 | Token Account | Associated Token Account（ATA） |
|---|---|---|
| 一言で | 特定のMintのトークンを保管する**汎用の財布** | ウォレット × Mint から**一意に導出される標準の財布** |
| アドレスの決まり方 | **任意のキーペア**（毎回ランダム） | **PDA**（mint, owner, tokenProgramから決定的に導出） |
| 同じowner × 同じMintで何個作れるか | **無数に作れる**（用途別に分けるのもアリ） | **1ウォレットにつき1個だけ**（一意） |
| 作成の手順 | ① `CreateAccount`（System）+ ② `InitializeAccount2`（Token-2022）の**2命令** | ① `CreateAssociatedToken`（ATA Program）の**1命令だけ** |
| アドレスを後から再計算できるか | **できない**（生成時のキーペアを覚えていないと分からない） | **できる**（mint + owner から `findAssociatedTokenPda` で常に再計算可能） |
| 一般的な用途 | DeFiの内部口座、複数口座を使い分けるサービス | **ユーザーがそのトークンを"普通に持っているとき"の口座** |

### 1.2 なぜ ATA という"標準"が後から定義されたのか

Solana 当初は **Token Account だけ**しかなく、「ユーザー A が USDC を持つには、A 用の Token Account を作る」という運用だった。しかしこの素朴な設計には**深刻な問題**があった:

> 「**A さんに USDC を送りたいんだけど、A さんの USDC 用 Token Account のアドレスって何？**」が**事前に分からない**。

A さんが Token Account を作るときに使ったキーペアはランダムで、A さん本人しか知らない。送金者が「A さんのウォレットアドレスは知ってるけど、A さんの USDC 口座のアドレスは知らない」という状況になり、**送金のたびに A さんに口座アドレスを教えてもらう**必要があった。Ethereum で言えば「同じ ERC20 でも宛先が毎回違う」ような状態。

これを解決するために導入されたのが **Associated Token Account Program** で、ルールは1つだけ:

> **ウォレットアドレス × Mint アドレス × Token Program から、PDA でアドレスを決定的に導出する**

これにより「A さんのウォレット = `xxx`、USDC のMint = `yyy`」さえ分かっていれば、**世界中の誰でも A さんの USDC ATA アドレスを計算で割り出せる**。送金者は A さんに何も聞かずに `findAssociatedTokenPda` で導出して送れる。

つまり ATA は「**ウォレット × Mint に対する標準的・規約的な Token Account**」であり、ユーザーが普通にトークンを保有するときは ATA を使うのが業界の事実上の標準になっている。

### 1.3 「ATA は Token Account の一種」という関係

混乱しやすいが整理すると:

```
Token Account（広い概念: Token-2022プログラムが管理する保管庫アカウント全般）
  ├─ 普通の Token Account（ランダムキーペアで作るタイプ） ← 今日の前半
  └─ Associated Token Account（PDA で導出するタイプ）   ← 今日の後半
```

**ATA も中身は Token Account と同じ**。違うのは **アドレスがどう決まるか** と、**作成命令がどのプログラム経由か**だけ。データレイアウトも、`owner`（=Token-2022）も、保有残高の記録方法も全部同じ。

### 1.4 ERC20 の USDC と Solana の USDC で比べる

Ethereum の ERC20 と比べると、Token Account と ATA の意味がかなり見えやすい。ここでは説明用に **USDC** を例にする。

> **注意**: 実際の USDC は Circle などが発行している既存トークンなので、自分が本物の USDC を勝手に作れるわけではない。ここでは「USDC という ERC20 / SPL Token がある」と仮定して、残高の持ち方だけを比較する。

#### ERC20 側

Ethereum の ERC20 では、**1つのコントラクトの中に「誰が何枚持っているか」の表**がある。

```
USDC ERC20 コントラクト
  残高表 balances
    Alice のウォレット     → 100 USDC
    Bob のウォレット       → 50 USDC
    あなたのウォレット    → 1000 USDC
```

つまり、あなたの USDC 残高を見るときは:

```
USDC ERC20 コントラクトの中の
balances[あなたのウォレット]
```

を見る。Ethereum では、**USDC コントラクトの内部に全員分の残高表がある**イメージでよい。

#### Solana 側

Solana では、この **「残高表の1行」** が、独立したアカウントになる。

```
USDC Mint
  Alice の USDC 用 Token Account     → 100 USDC
  Bob の USDC 用 Token Account       → 50 USDC
  あなたの USDC 用 Token Account    → 1000 USDC
```

この **「あなたの USDC 用 Token Account」** が、ERC20 でいう:

```
balances[あなたのウォレット]
```

に近い。

つまり Solana では、USDC Mint の中に全員分の残高表があるのではなく、**各人の USDC 残高が、それぞれ別々の Token Account に分かれている**。

#### ATA は何か

Solana では Token Account が独立したアドレスなので、次の問題が出る。

> あなたの USDC 残高は、どの Token Account アドレスにあるのか？

そこで標準ルールとして:

```
あなたのウォレット + USDC の Mint Address
→ あなたの USDC 用 ATA
```

という決まりがある。

つまり ATA は、**あなたの USDC 残高を置くための、標準的な Token Account** である。

まとめると:

```
Ethereum:
USDC コントラクト
  └─ balances[あなたのウォレット] = 1000

Solana:
USDC Mint
あなたの USDC 用 ATA
  └─ amount = 1000
```

**Token Account = ERC20 コントラクト内の「1人分の残高欄」を、Solana では独立したアドレスにしたもの。ATA = その中でも「ウォレット × Mint」から標準的に決まる Token Account。**

ここまで読むと「それなら ATA だけでよく、普通の Token Account はいつ必要なのか？」という疑問が出る。実務での使い分け、特に **ATA ではない Token Account を使う場面**については、後半の **「8. どう使い分けるのか」の実務的指針** で具体例つきで整理する。

### 1.5 Mint Account と Token Account と ATA の三者関係

ここまでで4種類の登場人物が出てきたので、図にしておく:

```
┌─────────────────────────────────────────────────────┐
│ Mint Account (mint.address)                         │
│   ・トークンの設計図                                │
│   ・decimals, mintAuthority, supply 等を保持        │
│   ・1トークンにつき 1 個                            │
└─────────────────────────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
        ▼              ▼              ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│Token Account │ │Token Account │ │     ATA      │
│ (任意の鍵)   │ │ (任意の鍵)   │ │ (PDA: 一意)  │
│ owner: Alice │ │ owner: Alice │ │ owner: Alice │
│ amount: 100  │ │ amount: 50   │ │ amount: 200  │
└──────────────┘ └──────────────┘ └──────────────┘
   ※ Aliceは複数のToken Accountを持ててもOK
   ※ ただしATAはAlice×Mintにつき1つだけ
```

「Mint = トークンの種類」「Token Account = そのトークンの口座」「ATA = ユーザー1人につき1つの標準口座」と覚える。

> **混同注意**: Token Account の中の `owner` フィールドはユーザーのウォレットアドレス（= 保有者）を指す。一方、Token Account 自身の **アカウントレベルの owner**（Solana ランタイムの owner）は **Token-2022 プログラム**を指す。前回（4/20）の 4.6〜4.7 で議論した「**システムレベルの owner（プログラム）** vs **データ内の owner フィールド（ユーザー）**」の2層構造が、Token Account でもそのまま現れる。

## 3. 追加するインポート

前回のインポートに、Token Account 用と ATA 用の関数を追加する。

```typescript
import {
  getInitializeMintInstruction,
  getInitializeAccount2Instruction,         // ★追加: Token Account 初期化
  getCreateAssociatedTokenInstructionAsync, // ★追加: ATA 作成
  getMintSize,
  getTokenSize,                             // ★追加: Token Account のサイズ取得
  TOKEN_2022_PROGRAM_ADDRESS,
  findAssociatedTokenPda,                   // ★追加: ATA アドレスの導出
} from "@solana-program/token-2022";
```

| 追加関数 | 役割 |
|---|---|
| `getInitializeAccount2Instruction` | 既に作った空アカウントを **Token Account として初期化**する命令を作る（旧版 `InitializeAccount` の改良版） |
| `getTokenSize` | Token Account の**データサイズ（バイト数）**を返す。Mint と Token Account はサイズが違う |
| `getCreateAssociatedTokenInstructionAsync` | ATA を**作成**する命令を作る。内部で PDA 導出 + System の CreateAccount + Token-2022 の InitializeAccount を1命令にまとめてくれる |
| `findAssociatedTokenPda` | ウォレット + Mint + Token Program から **ATA のアドレスを計算**する関数 |

> **`Account2` の "2" は何？**: Token-2022 では旧版命令 `InitializeAccount` の改良版として `InitializeAccount2` と `InitializeAccount3` が用意されている。違いは「**owner をアカウント引数として渡すか、命令データとして渡すか**」など些細なものだが、`Account2` は**owner を命令データに含めるので、署名アカウントを1つ減らせる**メリットがある。実用上は `Account2` を使えば困らない。

## 4. Token Account を作る（前半）

ここから前回のコードに**追記**していく。Mintの作成（前回 4.13まで）が終わった直後に、以下のコードを書く。

### 4.1 Token Account 用キーペアの生成

```typescript
const tokenAccount = await generateKeyPairSigner();
```

Token Account も**自分のアドレスを持つ独立したアカウント**なので、まずアドレスを決める必要がある。ここは Mint と全く同じパターンで、**ランダムなキーペアを生成**してそのアドレスを Token Account のアドレスにする。

> **重要な対比**: ATA の場合はこの行が不要になる。ATA は PDA で**自動的にアドレスが決まる**ので、わざわざキーペアを生成する必要がない。「ランダム vs 決定的」の差がここに現れる。

### 4.2 Token Account のサイズと Rent の計算

```typescript
const tokenAccountSpace = BigInt(getTokenSize());
const tokenAccountRent = await rpc
  .getMinimumBalanceForRentExemption(tokenAccountSpace)
  .send();
```

前回 Mint のために `getMintSize()` でサイズを取得したのと同じパターン。今回は **Token Account のサイズ**なので `getTokenSize()` を使う。

- **Mint Account のサイズ**: decimals, mintAuthority, supply, freezeAuthority などを含む
- **Token Account のサイズ**: mint, owner, amount, delegate, state, isNative, delegatedAmount, closeAuthority などを含む

両者は**全く別の構造**なので、サイズも別の関数で取る。

`getMinimumBalanceForRentExemption` は前回と同じ。**そのサイズのアカウントを永続的に存続させるために必要なSOL量**を RPC に問い合わせる。

> **Rentの再確認**: Token Account も**自分自身の lamports 残高で rent-exempt の保証金を保持する**。これは Mint と完全に同じ仕組み。後で `Close Token Account` 命令でこの SOL を回収できる。

### 4.3 命令①: System プログラムで空アカウントを作成

```typescript
const createTokenAccountInstruction = getCreateAccountInstruction({
  payer: feePayer,
  newAccount: tokenAccount,
  lamports: tokenAccountRent,
  space: tokenAccountSpace,
  programAddress: TOKEN_2022_PROGRAM_ADDRESS,
});
```

これは前回 Mint を作るときに使った `getCreateAccountInstruction` と**完全に同じパターン**。引数の意味も前回 4.6 で詳説した通り:

- **`payer`**: rent と手数料を払うのは feePayer
- **`newAccount`**: 作る先（= 今回は `tokenAccount`）
- **`lamports`**: 初期残高（= rent-exempt 額）
- **`space`**: データ領域サイズ（= `getTokenSize()` の値）
- **`programAddress: TOKEN_2022_PROGRAM_ADDRESS`**: 作ったあとのオーナー（= 書き換え担当プログラム）。Token Account も Token-2022 が書き換えるので Mint と同じく Token-2022 を指定

つまり**Solanaにおけるアカウント作成は常に同じ手順**で、「サイズ」と「rent」と「オーナーにするプログラム」だけ変えれば、何のアカウントでも作れる。**Token-2022 プログラムが所有する空アカウント**ができたが、まだ「これは Token Account です」とは認識されていない。

### 4.4 命令②: Token-2022 プログラムで Token Account を初期化

```typescript
const initializeTokenAccountInstruction = getInitializeAccount2Instruction({
  account: tokenAccount.address,
  mint: mint.address,
  owner: feePayer.address,
});
```

前のステップで作った「空のアカウント」を、Token-2022 プログラムが読んで**Token Account としての中身を書き込む**命令。

引数の意味:

- **`account: tokenAccount.address`**: 初期化対象の Token Account アドレス
- **`mint: mint.address`**: **どの Mint のトークンを保管するか**を指定。Token Account は必ず1つの Mint に紐づく（USDC用のToken AccountにDAIは入れられない）
- **`owner: feePayer.address`**: **このToken Accountに入っているトークンを使える人**のウォレットアドレス。今回は feePayer 自身が保有者になる

> **ここの `owner` は前回（4/20）4.6 で議論した2層構造の "データ内の owner フィールド"** に該当する。アカウントレベルのowner（= Token-2022プログラム）は3.3の `getCreateAccountInstruction` で `programAddress` として既に指定済み。**同じ "owner" という単語が、一方ではプログラム、一方ではユーザーを指している**点に再注意。

### 4.5 命令を配列にまとめてトランザクションを送信

```typescript
const tokenAccountInstructions = [
  createTokenAccountInstruction,
  initializeTokenAccountInstruction,
];

const { value: createTokenAccountLatestBlockhash } = await rpc
  .getLatestBlockhash()
  .send();

const tokenAccountTxMessage = pipe(
  createTransactionMessage({ version: 0 }),
  (tx) => setTransactionMessageFeePayerSigner(feePayer, tx),
  (tx) =>
    setTransactionMessageLifetimeUsingBlockhash(
      createTokenAccountLatestBlockhash,
      tx,
    ),
  (tx) => appendTransactionMessageInstructions(tokenAccountInstructions, tx),
);

const signedTokenAccountTxMessage = await signTransactionMessageWithSigners(
  tokenAccountTxMessage,
);

const signedTokenAccountTxMessageWithLifetime =
  signedTokenAccountTxMessage as typeof signedTokenAccountTxMessage & {
    lifetimeConstraint: {
      lastValidBlockHeight: bigint;
    };
  };

await sendAndConfirmTransactionFactory({ rpc, rpcSubscriptions })(
  signedTokenAccountTxMessageWithLifetime,
  { commitment: "confirmed" },
);

const tokenAccountTxSignature = getSignatureFromTransaction(
  signedTokenAccountTxMessageWithLifetime,
);
```

ここから先は**前回の Mint 作成のときと完全に同じパターンの繰り返し**である。

1. **2つの命令を配列にまとめる** — System の CreateAccount と Token-2022 の InitializeAccount2 を1トランザクションに入れることで、**アトミック（両方成功 or 両方失敗）**を保証する。空アカウントだけ作って Token Account として初期化されないまま、という中途半端な状態を防ぐ。
2. **最新の blockhash を取る** — トランザクションの有効期限の基準にする。
3. **`pipe` で TransactionMessage を組み立てる** — 「空のメッセージ → 支払者をセット → 有効期限をセット → 命令を追加」の関数型パイプライン。
4. **署名する** — `feePayer` と `tokenAccount` の両方の秘密鍵で署名（CreateAccount の `newAccount` も署名者として必要、という前回 4.6 で議論した「本人同意の署名」ルール）。
5. **型アサーションで `lifetimeConstraint` を補う** — 前回 4.12 と同じ、`@solana/kit` の型定義のクセを吸収するおまじない。
6. **`sendAndConfirmTransaction` で送信・確認** — Factory パターンで rpc/ws を渡し、即座にトランザクションを実行。`confirmed` まで待つ。
7. **署名（Signature）を取り出す** — エクスプローラで検索可能なTX一意IDをログ用に取得。

これで **Token Account が完成**し、`tokenAccount.address` が「feePayer が `mint.address` のトークンを保管できる口座のアドレス」として機能するようになる。

```typescript

console.log("Transaction Signature:", transactionSignature);
console.log("\nToken Account Address:", tokenAccount.address);
console.log("Token Account Transaction Signature:", tokenAccountTxSignature);
```

**Token Account のアドレス（`3Qbdmg...`）はランダム生成されたキーペア由来**なので、実行するたびに毎回違うアドレスになる。これが「**送金者は事前にこのアドレスを知ることができない**」という1.2で議論した問題の正体である。

## 5. Associated Token Account を作る（後半）

ここから **ATA 編**。同じスクリプトの続きとして、Token Account 作成のあとに追記する。

### 5.1 ATA アドレスを PDA で導出

```typescript
const [associatedTokenAccountAddress] = await findAssociatedTokenPda({
  mint: mint.address,
  owner: feePayer.address,
  tokenProgram: TOKEN_2022_PROGRAM_ADDRESS,
});

console.log(
  "\nAssociated Token Account Address:",
  associatedTokenAccountAddress,
);
```

ここが **Token Account との最大の違い**。Token Account のときは `generateKeyPairSigner()` でランダムなアドレスを作っていたが、ATA は **PDA（Program Derived Address）** で**決定的に導出**する。

`findAssociatedTokenPda` の引数:

- **`mint: mint.address`**: どの Mint に対する ATA か
- **`owner: feePayer.address`**: ATA の保有者となるウォレット
- **`tokenProgram: TOKEN_2022_PROGRAM_ADDRESS`**: どの Token プログラム（旧 SPL Token か Token-2022 か）の ATA か

この3つの値から、Associated Token Account Program のアドレスと組み合わせて**ハッシュ計算**でアドレスが決まる。**全く同じ3つの引数を渡せば、いつどこで誰が呼んでも同じアドレスが返ってくる**。

> **戻り値が配列なのは？**: `findAssociatedTokenPda` は `[address, bump]` のタプルを返す。`bump` は PDA 計算時に Ed25519 曲線上にない点を見つけるための追加バイト（0〜255）で、ATA を作成する命令の内部で使われる。今回はアドレスだけ使うので分割代入で `bump` は捨てている。

> **PDAについての復習**: 前回（4/20）4.6 の "feePayer ≒ EOA" の議論の中で、「Solanaには Keypair 方式と PDA 方式の2種類のアドレスがあり、ATA は PDA の典型例」と予告していた。今日その**実物**に初めて触れたことになる。

#### 「ATA はまだ作られていないのに、なぜアドレスが分かるのか」

ここが ATA の**最も革命的な点**である。**まだチェーン上に存在していないアカウントのアドレスを、計算だけで先に求められる**。

```
[mint.address] + [feePayer.address] + [TOKEN_2022_PROGRAM_ADDRESS] + [ATA Program ID]
                          ↓
                      ハッシュ計算
                          ↓
              [ATA Address]  ← まだ作られていなくても求まる
```

これにより以下のようなことが可能になる:

- **送金者が受信者のATAアドレスを事前計算して送金できる**（受信者がまだATAを作っていなくても、送金時に作ってあげるパターンが定石）
- **ウォレットUIが「あなたのUSDC残高は X です」を、ATAを開かなくても自動で計算できる**（毎回 PDA 導出して残高をチェック）
- **チェーン上のスキャンが容易**（誰のどのMintのATAかが、アドレスから逆算できる）

### 5.2 ATA 作成命令を組み立てる

```typescript
const createAtaInstruction = await getCreateAssociatedTokenInstructionAsync({
  payer: feePayer,
  mint: mint.address,
  owner: feePayer.address,
});
```

ここが Token Account との**第二の大きな違い**。Token Account のときは:

```
①CreateAccount（System） + ②InitializeAccount2（Token-2022）= 2命令
```

の2ステップが必要だったが、ATA は:

```
CreateAssociatedToken（ATA Program）= 1命令
```

これだけで済む。Associated Token Account Program が**内部で System の CreateAccount と Token-2022 の InitializeAccount を CPI（Cross-Program Invocation）で呼んでくれる**ので、クライアント側はラッパー命令1つを呼ぶだけになる。

引数の意味:

- **`payer: feePayer`**: rent と手数料の支払者
- **`mint: mint.address`**: どの Mint の ATA を作るか
- **`owner: feePayer.address`**: ATA の保有者

> **`Async` 接尾辞は何？**: 関数名末尾の `Async` は「**内部で `findAssociatedTokenPda` を非同期に呼んで PDA を導出する**」という意味。同期版の `getCreateAssociatedTokenInstruction`（PDA を引数で渡す形）もあるが、`Async` 版の方が「ライブラリに任せておけば導出してくれる」のでミスが起きにくく便利。

### 5.3 トランザクションを組み立てて送信

```typescript
const { value: createAtaLatestBlockhash } = await rpc
  .getLatestBlockhash()
  .send();

const ataTxMessage = pipe(
  createTransactionMessage({ version: 0 }),
  (tx) => setTransactionMessageFeePayerSigner(feePayer, tx),
  (tx) =>
    setTransactionMessageLifetimeUsingBlockhash(createAtaLatestBlockhash, tx),
  (tx) => appendTransactionMessageInstructions([createAtaInstruction], tx),
);

const signedAtaTxMessage =
  await signTransactionMessageWithSigners(ataTxMessage);

const signedAtaTxMessageWithLifetime =
  signedAtaTxMessage as typeof signedAtaTxMessage & {
    lifetimeConstraint: {
      lastValidBlockHeight: bigint;
    };
  };

await sendAndConfirmTransactionFactory({ rpc, rpcSubscriptions })(
  signedAtaTxMessageWithLifetime,
  { commitment: "confirmed" },
);

const ataTxSignature = getSignatureFromTransaction(
  signedAtaTxMessageWithLifetime,
);

console.log(
  "\nAssociated Token Account Creation Transaction Signature:",
  ataTxSignature,
);
```

ここは**前回も今日の Token Account 作成も同じ**お決まりのパターン。1点だけ違うのは:

- **`appendTransactionMessageInstructions([createAtaInstruction], tx)`** — 命令が**1つしか入っていない**

Token Account 作成時は `[createTokenAccountInstruction, initializeTokenAccountInstruction]` の2命令だったが、ATA 作成は1命令で済むのでシンプルになる。

> **署名者が減る**: Token Account 作成時は `[feePayer, tokenAccount]` の2人の署名が必要だった（CreateAccount の newAccount 規約により）。ATA 作成では **`feePayer` の署名だけで OK**。なぜなら ATA のアドレスは PDA で**秘密鍵が存在しない**ので、本人同意の署名を提供できないからである。代わりに Associated Token Account Program が**`invoke_signed` で seed を使った特殊な署名**を内部で行うことで、Solana ランタイムの「新規アカウントは署名者でなければならない」ルールを満たしている。

注目ポイント:

- **Token Account Address (`HNDtCe...`)** と **ATA Address (`Gkx4Lg...`)** は**同じスクリプト実行内**でも**別々のアドレス**になっている — 同じ feePayer × 同じ Mint に対して、両方のタイプの口座が共存している
- ATA Address は `mint.address` と `feePayer.address` から決定的に導出されるので、**スクリプトを再実行しても同じ Mint と同じ feePayer なら毎回同じアドレスになる**（ただし feePayer は毎回新規生成しているので実際は変わる）
- **Mint が変わると当然 ATA アドレスも変わる**（PDA の seed が変わるため）


## 7. Token Account vs ATA: 命令フローの比較図

両者の違いを「**何の命令を、どのプログラムに送るか**」のレベルで対比する。

### Token Account 作成

```
TypeScriptクライアント
    │
    ├─ 命令①: CreateAccount → System Program
    │     ・新しい空アカウントを作る
    │     ・owner = TOKEN_2022_PROGRAM_ADDRESS
    │     ・space = getTokenSize()
    │     ・lamports = rent-exempt 額
    │
    └─ 命令②: InitializeAccount2 → Token-2022 Program
          ・mint, owner（=ユーザー）を中身に書き込む

  ※ 1トランザクション内に2命令、署名者は feePayer + tokenAccount
```

### ATA 作成

```
TypeScriptクライアント
    │
    └─ 命令: CreateAssociatedToken → Associated Token Account Program
          │
          ├─ 内部 CPI 呼び出し → System Program (CreateAccount)
          │   ・PDA アドレスに空アカウントを作る
          │   ・invoke_signed で ATA Program が署名肩代わり
          │
          └─ 内部 CPI 呼び出し → Token-2022 Program (InitializeAccount)
              ・mint, owner を書き込む

  ※ 1トランザクション内に1命令、署名者は feePayer のみ
```

クライアントから見ると **ATA の方が圧倒的にシンプル**だが、**チェーン側でやっていることは実質同じ**。違いは「クライアントが2命令を組み立てる」か「ATA Program が1命令の中で2命令分の処理を CPI でやる」かだけ。

## 8. 「どう使い分けるのか」の実務的指針

ここまで違いを整理したので、**実際の開発では何を選ぶか**のガイドラインをまとめる。

### 8.1 ユーザーが普通にトークンを保有する用途 → ATA

- ウォレット内で USDC を保有する、ゲーム内通貨を持つ、NFT を保管する、等
- **「あの人にトークンを送りたい」が成立する場面では基本ATA**
- ウォレットUI（Phantom, Solflare等）も基本 ATA を表示する
- Solana エコシステムの DApp（Jupiter, Raydium, Marinade等）が前提とするのも ATA

自分でトークンを作って、それを自分のウォレットで受け取るだけなら、基本的には **ATA だけで十分**。この場合は:

```
Mint Address を作る
あなたのウォレット用の ATA を作る
その ATA に mint する
```

という流れになる。つまり「自分のウォレットが普通にそのトークンを持つ」だけなら、ランダムキーペアで別の Token Account を作る必要はほとんどない。

### 8.2 内部処理用の口座、複数口座を使い分けたい用途 → 普通の Token Account

では、ATA ではない普通の Token Account はいつ使うのか。代表例は、**ウォレットではなくプログラムにトークンを預けたいとき**である。

例えば:

- DEX の流動性プール
- ステーキング金庫
- エスクロー
- NFT 販売の一時預かり
- ゲーム内の報酬プール

こういう場合、トークンは「あなた個人のウォレットの標準口座」に置くのではなく、**プログラムが管理する専用の Token Account** に置く。

ここでいう「プログラムが管理する専用の Token Account」は、Solana 全体で共通の口座を使うという意味ではない。通常は **プログラムごと・用途ごと・Mint ごとに別々の Token Account** を持つ。

```
あなたのウォレット
  └─ あなたの USDC 用 ATA

ステーキングプログラム
  └─ 預かり用 USDC Token Account

DEX A
  └─ USDC プール用 Token Account

DEX B
  └─ USDC プール用 Token Account
```

この「預かり用 USDC Token Account」や「USDC プール用 Token Account」は、あなたのウォレット用 ATA ではない。プログラム側のルールに従ってだけ動かせる、専用の保管庫として使われる。

その他にも、普通の Token Account は以下のような場面で使われる:

- DeFi プロトコルの内部勘定（プールごとの預かり金、手数料の一時口座、等）
- 取引所の Hot Wallet / Cold Wallet
- 1ユーザーに対して「目的別に複数の口座」を持たせたいケース
- プログラムが PDA で自前の Token Account を持つケース（= プログラム所有のVault的な口座）

### 8.3 「迷ったら ATA」が現代の答え

ATA が後から導入されたという経緯から、Solana の現代的なベストプラクティスは:

> **ユーザーがトークンを保有する目的なら、迷わず ATA**

実務でランダムキーペア方式の Token Account を新規に作るのは、上記の特殊用途のときだけ。**今日 Token Account の作り方を学んだのは、原理を理解するため**であって、普段は ATA を使うことになる。

## 9. 今回学んだキーコンセプト

1. **Token Account は「特定のMintのトークンを保管する財布」**
   - 1つの Mint に紐づく
   - 自分自身の rent-exempt 用 lamports を持つ独立アカウント
   - データ内に `owner` フィールド（= 保有者ウォレット）と `amount` フィールド（= 残高）を持つ
   - アカウントレベルの owner（書き換え担当プログラム）は Token-2022

2. **ATA は「ウォレット × Mint から PDA で一意に導出される標準 Token Account」**
   - 同じ owner × Mint × Token Program に対して**世界に1つだけ**
   - アドレスは事前計算可能なので**送金者が宛先を知らなくても送れる**
   - 中身のデータレイアウトは Token Account と同じ
   - 違うのは「アドレスの決まり方」と「作成命令の数」だけ

3. **作成手順の違い**
   - Token Account: System の CreateAccount + Token-2022 の InitializeAccount2 の**2命令**
   - ATA: Associated Token Account Program の CreateAssociatedToken の**1命令**（中で CPI）

4. **署名者の違い**
   - Token Account 作成: `feePayer` と `tokenAccount`（newAccount）の2署名
   - ATA 作成: `feePayer` のみ（PDA は秘密鍵を持たないので、ATA Program が `invoke_signed` で署名肩代わり）

5. **`@solana/kit` の関数型パターンは引き続き同じ**
   - `pipe()` でメッセージ組み立て、Factory パターンで送信、はどのアカウント作成でも共通
   - Mint・Token Account・ATA で「やっていることの本質」が違うだけで、コードの**外形**はかなり似ている

## 10. 前回（4/20）からの積み上がり

| 項目 | 4/20（Mint 作成） | 4/28（Token Account / ATA 作成） |
|---|---|---|
| 主役のアカウント | Mint Account | Token Account / Associated Token Account |
| サイズ取得関数 | `getMintSize()` | `getTokenSize()` |

## 12. まとめ

今回の学習で、Solana 上の **トークン保有口座**の2つの作り方を実装した。

- **Token Account**: 任意のキーペアで作る、汎用の保管庫。System の CreateAccount + Token-2022 の InitializeAccount2 の2命令で作成。
- **Associated Token Account（ATA）**: ウォレット × Mint から PDA で一意に導出される、標準の保管庫。Associated Token Account Program の1命令で作成。

両者の違いの本質は **「アドレスの決まり方」**：

> **任意（Keypair） vs 決定的（PDA）**

この差から、**ATA は宛先を事前に計算できる** = **送金がスケールする**、というユーザー体験の違いが生まれている。実務では**ユーザー保有のための口座は ATA を使う**のが現代のベストプラクティスであり、Token Account（ランダム方式）は内部勘定や特殊用途にとどまる。

```
Mint（トークン設計図 = 4/20で作成）
   ↓
Token Account / ATA（トークン保管庫 = 今日作成）
   ↓
???（次回: 実際にトークンを発行・送金）
```

次回以降は、今日作った口座に**実際にトークンを発行（Mint）**し、**別の口座へ送る（Transfer）**、**焼却する（Burn）**、**口座を閉じて rent を回収する（Close）**、といったライフサイクル操作に入っていく予定。Mint・口座・残高操作の3点セットが揃うと、SPL トークンの実用的な機能はほぼ全て触れることになる。

前回も書いたが、`@solana/kit` の関数型・Factory パターンの**お作法**が頭に入ってくると、新しいアカウントタイプの作り方も**ほとんど同じテンプレート**で書けるようになる。今日の Token Account 作成は構造的には Mint 作成のコピペに近かったし、ATA はそれをさらに簡略化した形。**繰り返し書くことで Solana 開発のリズムが身につく**実感がある。
