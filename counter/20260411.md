## 1. 学習の全体像

本日の学習の最大のポイントは、**Solana公式のテンプレートをそのまま使ってDAppを立ち上げた**ことである。

これまでは以下のように、全てを手作業で構築していた:

- `anchor init` でプログラムだけを作り、
- `create-react-app` でフロントエンドを別途立ち上げ、
- Phantom Walletや `@coral-xyz/anchor` を手で配線する

これに対して、Solana公式が提供する `create-solana-dapp` コマンドを使うと、以下が**一発で全部揃った状態**でプロジェクトが生成される:

- Anchorプログラム（Rustのスマートコントラクト）
- Next.js + Tailwind CSS のフロントエンド
- Wallet Adapter（Phantomだけでなく複数ウォレットに対応）
- `@tanstack/react-query` による状態管理
- プログラムとフロントエンドを繋ぐ型付きクライアント
- テスト用のコード

つまり、**配線済みの雛形が最初から用意されている**状態である。これは前回3/15の学習で手作業で組み立てたものを、公式が「ベストプラクティス込み」でまとめてくれたものに相当する。今日はこのテンプレートを、実際に触って動かしてみた。

そしてテンプレートの中身を理解するために、最後に「2ずつ増やすカウンター機能」を自分で追加してみた。

## 2. Solana公式テンプレートでプロジェクトを作る

### 2.1 利用可能なテンプレート一覧を見る

```bash
npx create-solana-dapp --list-templates
```

`npx` は「npmパッケージを一時的にダウンロードして実行する」コマンド。インストール不要で最新版のツールを使える。`--list-templates` オプションを付けると、Solana公式が用意しているテンプレートの一覧が表示される。

ここで**テンプレート名を確認**してから、次のステップで指定する。

### 2.2 テンプレートからプロジェクトを生成

```bash
npx create-solana-dapp -t web3js-next-tailwind-counter
```

- `-t` は `--template` の省略形。使うテンプレートを指定する。
- `web3js-next-tailwind-counter` は「カウンターDApp」の雛形。名前の通り:
  - **web3js**: Solana JavaScriptライブラリを利用
  - **next**: Next.js（Reactのフレームワーク）をフロントエンドに採用
  - **tailwind**: Tailwind CSSでスタイリング
  - **counter**: カウンターのサンプルアプリ

このコマンドを実行すると、プロジェクトディレクトリが生成され、以下のような構造が自動で出来上がる:

```
counter/
├── anchor/                    ← スマートコントラクト (Rust)
│   ├── programs/counter/src/lib.rs
│   └── Anchor.toml
├── src/                       ← フロントエンド (Next.js + React)
│   └── components/counter/
│       ├── counter-data-access.tsx  ← プログラム呼び出しロジック
│       ├── counter-ui.tsx           ← UI部品
│       └── counter-feature.tsx      ← 画面全体
├── package.json
└── ...
```

> **強調ポイント**: 前回（3/15）は Reactのプロジェクトを `create-react-app` で作り、そこに `@solana/web3.js` や `@coral-xyz/anchor` を後から手動で追加し、Phantom Walletの接続処理まで全部自分で書いた。今回のテンプレートは、**それらの面倒を全部肩代わりしてくれている**。Anchor v0.30+ のIDL形式にも最初から対応しており、「バージョン不整合でハマる」というトラブルも起きない。

## 3. プログラムのビルドとLocalnetでの動作確認

### 3.1 プログラムのビルド

```bash
cd anchor
anchor build
```

`anchor build` は、Rustで書かれたスマートコントラクトをコンパイルして**バイトコード（Solanaが実行できる形式）**を生成するだけのコマンド。ブロックチェーンへのアップロードはまだ行わない。

ビルド結果は `anchor/target/` 配下に出力される:

- `target/deploy/counter.so`: デプロイ用のバイトコード
- `target/idl/counter.json`: IDL（プログラムの取扱説明書）
- `target/types/counter.ts`: TypeScript用の型定義

### 3.2 ローカルチェーンの起動（Solana Test Validator）

```bash
# 別のターミナルを開いて実行
cd anchor
solana-test-validator
```

`solana-test-validator` は、**自分のPCの中にSolanaブロックチェーンを1つ立ち上げる**コマンド。これを「Localnet」と呼ぶ。

Localnetを使うと:

- インターネット接続なしで開発・テストできる
- 何度でも無料で(擬似)SOLがもらえる
- 失敗してもやり直せる（チェーンを削除して再起動すればよい）

devnet（Solanaが提供するテストネット）よりもさらに手元寄りの、開発者専用の遊び場である。

このターミナルは起動したまま放置する（止めるとチェーンも止まる）。

### 3.3 プログラムのデプロイ

元のターミナルに戻って:

```bash
anchor deploy
```

`anchor deploy` は内部的に以下の**4ステップ**を実行している:

1. **プログラムをブロックチェーンにアップロード** — バイトコードをチェーン上に書き込む
2. **アカウントを作成** — プログラムを置くための「プログラムアカウント」を作る
3. **トランザクションを送信** — デプロイに必要な命令をチェーンに送る
4. **プログラムIDを取得** — デプロイ先のアドレス（Public Key）が決まる

### 3.4 プログラムIDの設定（エラー対処）

初回の `anchor deploy` 後にテストを走らせると、プログラムIDの不整合エラーが出る。これは、ソースコード側に書かれた `declare_id!(...)` のプログラムIDと、実際にチェーンにデプロイされたIDが一致していないため。

以下の2箇所を、`anchor deploy` が表示したプログラムIDで書き換える:

- `anchor/programs/counter/src/lib.rs` の `declare_id!("...")`
- `anchor/Anchor.toml` の `[programs.localnet]` セクション

書き換えたら、**もう一度ビルドとデプロイ**する:

```bash
anchor build
anchor deploy
```

> **補足**: このステップは Solana / Anchor 開発でよく知られた"儀式"である。テンプレートのダミーIDをそのままにしておくと、最初のデプロイ時にIDが自動で差し替わるため、ソース側と合わなくなる。

### 3.5 テスト

```bash
anchor test --skip-local-validator
```

`anchor test` は通常、内部で自動的に `solana-test-validator` を立ち上げてテストを走らせるが、今回は既に別ターミナルで Validator を起動済みなので、`--skip-local-validator` オプションを付けて「Validatorの起動をスキップ」してもらう。

テンプレートに最初から含まれているテストが走り、プログラムが正しく動いていることを確認できる。

## 4. フロントエンドの起動とPhantom Walletの接続

### 4.1 フロントエンドの起動

```bash
# さらに新しいターミナルで実行
npm run dev
```

これで Next.js の開発サーバーが立ち上がり、ブラウザで `http://localhost:3000` を開くとフロントエンドが表示される。

### 4.2 Phantom Walletを Localnet に繋ぐ

Localnet上のプログラムと通信するには、Phantom Wallet側でも接続先を Localnet にする必要がある:

1. Phantom Wallet → 設定 → 開発者設定
2. テストネットモードを有効化
3. **Solana Localnet** を選択

これで、ブラウザのPhantomが自分のPC上の `solana-test-validator` に繋がる状態になる。

### 4.3 ウォレットに擬似SOLを投入

Localnet上のウォレットには最初SOLが1つも入っていないので、トランザクション手数料を払えない。以下のコマンドで自分のアドレスに2 SOL送り込む:

```bash
# 新しいターミナルで
solana airdrop 2 <自分のウォレットアドレス>
```

`<自分のウォレットアドレス>` は Phantom Walletの画面上部のアドレスをコピペする。Localnetなので、SOLは好きなだけ無料で airdrop できる。

### 4.4 カウンターの動作確認

フロントエンドの「Counter Program」タブを開き、以下を順に試す:

1. **Create** — 新しいカウンターアカウントを作成（Phantomで承認）
2. **Increment** — カウンターを1増やす
3. **Decrement** — カウンターを1減らす
4. **Set** — カウンター値を任意の数に設定

全ての操作でPhantomの承認ポップアップが出て、処理後に画面のCount数値が更新されれば成功である。

## 5. 「2ずつ増やす」機能を追加する

テンプレートの動作確認ができたので、自分で機能を追加してテンプレートの構造を理解する。題材は **「Increment by 2」（2ずつ増やすボタン）** とした。

追加が必要な箇所は3つ:

1. **スマートコントラクト側**（Rust）: 新しい命令 `increment_by_two` を追加
2. **フロントエンドのデータアクセス層**（TypeScript）: `incrementByTwoMutation` を追加
3. **フロントエンドのUI層**（TypeScript/JSX）: ボタンを追加

### 5.1 スマートコントラクト側の修正

`anchor/programs/counter/src/lib.rs` に以下を追加する:

```rust
pub fn increment_by_two(ctx: Context<Update>) -> Result<()> {
    ctx.accounts.counter.count = ctx.accounts.counter.count.checked_add(2).unwrap();
    Ok(())
}
```

既存の `increment` 関数とほぼ同じで、`checked_add(1)` を `checked_add(2)` に変えているだけ。

- **`checked_add`**: オーバーフロー（型の最大値を超える加算）が起きた場合に `None` を返す安全な加算。`unwrap()` は「成功してるはず」として中身を取り出すが、オーバーフロー時にはパニックする。
- **`Context<Update>`**: 既存の `increment` と同じく `Update` アカウント構造体を流用する（カウンターアカウントを `mut` で受け取るだけ）。新しい構造体を定義する必要はない。

修正後は**必ず再ビルド・再デプロイ**する:

```bash
cd anchor
anchor build
anchor deploy
```

さらにIDL（`anchor/target/idl/counter.json`）と型定義（`anchor/target/types/counter.ts`）にも `incrementByTwo` が自動で追加されているはずである。これらはフロントエンドから参照されるので、ビルドを忘れるとフロント側で新しい命令が見えない。

### 5.2 フロントエンドのデータアクセス層の修正

`src/components/counter/counter-data-access.tsx` に `incrementByTwoMutation` を追加する。場所は `incrementMutation` のすぐ下が分かりやすい:

```typescript
const incrementByTwoMutation = useMutation({
  mutationKey: ['counter', 'increment_by_two', { cluster, account }],
  mutationFn: () => program.methods.incrementByTwo().accounts({ counter: account }).rpc(),
  onSuccess: async (tx) => {
    transactionToast(tx)
    await accountQuery.refetch()
  },
})
```

さらに、`useCounterProgramAccount` フックの `return { ... }` に `incrementByTwoMutation` を追加する。これを忘れるとUI側から呼び出せない。

**重要な命名ルール**: Rust側で `increment_by_two`（スネークケース）と書いた命令は、フロントエンド側では `program.methods.incrementByTwo()`（キャメルケース）になる。これは `@coral-xyz/anchor` が自動変換してくれる仕様。前回（3/15）に学んだ `amount_donated` → `amountDonated` と同じ原理である。

**`useMutation` とは**: `@tanstack/react-query` が提供する「書き込み操作（=状態を変えるリクエスト）」用のフック。`mutateAsync()` でトランザクションを発火し、`isPending` でローディング状態を取得できる。成功時に `onSuccess` が呼ばれ、そこで `accountQuery.refetch()` を呼ぶことで画面のカウンター表示を最新化している。

> **補足**: 前回（3/15）は `useState` で自力で状態管理していたが、テンプレートは `@tanstack/react-query` を使っている。これは Web2フロントエンド界隈で広く使われている定番ライブラリで、「サーバー（=今回はブロックチェーン）からのデータ取得・キャッシュ・再取得」を面倒みてくれる。Solana公式がこれを採用しているということは、**React + Solana のベストプラクティスはこれ**ということだろう。

### 5.3 フロントエンドUI層の修正

`src/components/counter/counter-ui.tsx` の `CounterCard` コンポーネント内、既存の Increment ボタンの直後に「Increment by 2」ボタンを追加する:

```tsx
<Button
  variant="outline"
  onClick={() => incrementByTwoMutation.mutateAsync()}
  disabled={incrementByTwoMutation.isPending}
>
  Increment by 2
</Button>
```

加えて、コンポーネント冒頭の `useCounterProgramAccount` の分割代入に `incrementByTwoMutation` を加えるのも忘れないこと:

```tsx
const { accountQuery, incrementMutation, incrementByTwoMutation, setMutation, decrementMutation, closeMutation } = useCounterProgramAccount({
  account,
})
```

### 5.4 動作確認

フロントエンドはホットリロード（保存すると自動で画面が更新される）されるので、追加したボタンを押してカウンターが2増えればOK。Phantomの承認 → カウンター更新、という一連の流れが問題なく動く。

## 6. 開発中に遭遇したエラーと対処法

| エラー / 現象 | 原因 | 対処法 |
|---|---|---|
| `anchor test` でプログラムIDの不整合エラー | 初回デプロイでIDが自動生成され、ソース側のダミーIDと合わなくなった | `lib.rs` と `Anchor.toml` を新しいIDで書き換えて、再ビルド・再デプロイ |
| フロントから Increment by 2 が呼べない（型エラー） | `anchor build` を忘れてIDL/型定義が古いまま | `cd anchor && anchor build` で型を再生成 |
| ボタンが2つとも "Increment" 表示になる | UI追加時にラベルをコピペしたまま変更し忘れた | 2つ目のボタンのラベルを "Increment by 2" に修正 |
| `npm run dev` がワークスペース警告を出す | ホームディレクトリ `/home/kenichiuejo/` に古い `package.json` と `package-lock.json` が残っており、Next.js 16のTurbopackが誤ってそこをワークスペースルートとして検出した | ホームの不要な `package.json` / `package-lock.json` / `node_modules` を削除（他プロジェクトに影響がないことを先に確認） |
| Phantomで署名できない | Phantomの接続先が devnet のままで Localnet に切り替え忘れ | Phantom → 設定 → 開発者設定 → Solana Localnet を選択 |
| トランザクションが "insufficient funds" | Localnetウォレットに SOL が無い | `solana airdrop 2 <アドレス>` で補充 |

## 7. 前回（3/15）からの学びの進化

3/15の Crowdfunding DApp と、今日の Counter DApp を比較することで、**「手作り」と「テンプレート利用」の違い**が明確になった。

| 項目 | 3/15 (手作り) | 4/11 (テンプレート) |
|---|---|---|
| プロジェクト生成 | `anchor init` + `create-react-app` | `npx create-solana-dapp -t web3js-next-tailwind-counter` 一発 |
| フロントフレームワーク | Create React App (CRA) | **Next.js 16 + Turbopack** |
| スタイリング | 素の CSS | **Tailwind CSS + shadcn/ui** |
| 状態管理 | `useState` / `useEffect` 自作 | **`@tanstack/react-query` でキャッシュ付き** |
| ウォレット接続 | `window.solana` を直接叩く | **Wallet Adapter（複数ウォレット対応）** |
| プログラム呼び出し | 1つの App.js にベタ書き | **data-access層 と UI層 に分離** |
| チェーン | devnet | **Localnet (solana-test-validator)** |
| IDLのコピー | 手動で `cp target/idl/xxx.json src/idl.json` | **テンプレートが自動で配線済み** |
| TypeScript型 | なし（JavaScript） | **プログラムの型が自動生成される** |

これを見ると、Solana公式テンプレートがどれほど「本番開発のベストプラクティス」をまとめ上げてくれているかが一目瞭然である。最初から手作りで組むと時間がかかるうえ、どこで手を抜いていいか分からない部分まで自作してしまいがちだが、テンプレートは**「プロが書いた標準構成」をそのまま出発点**にできる。

## 8. まとめ

今回の学習では、Solana公式の **`create-solana-dapp` テンプレート**を使って、カウンターDAppを以下の流れで体験した:

```
[テンプレート生成] → [ビルド] → [Localnet起動] → [デプロイ] →
[プログラムID修正] → [再デプロイ] → [テスト] → [フロント起動] →
[Phantom接続] → [airdrop] → [動作確認] → [機能追加(increment_by_two)]
```

学びの核心は3つ:

1. **Solana公式テンプレートを積極的に使うべき** — 手作りで時間をかけるより、テンプレートの中身を読んで理解するほうが学習効率が圧倒的に高い。特に `web3js-next-tailwind-counter` は Next.js + Tailwind + Wallet Adapter + React Query という現代的な構成で、そのまま本番開発にも使えるレベル。
2. **Localnet（solana-test-validator）は開発のベストフレンド** — devnet より速く、無料で何度でも試せる。プログラムを書いて壊して作り直す、という試行錯誤に最適。
3. **テンプレートでも"プログラムID書き換え儀式"は必要** — ダミーIDが残っている場合は、最初のデプロイ後に `lib.rs` と `Anchor.toml` を書き換えて再デプロイする手順を忘れない。

そして冒頭にも書いた通り、**次回からはSolana公式の日本語Bootcamp動画（https://www.youtube.com/watch?v=kOjNz61UzA4&t=1323s）を軸に学習を進める**。今日のテンプレート体験は、その準備運動として非常に良い導入になった。
